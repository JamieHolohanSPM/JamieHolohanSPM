Curam TI CDEJ
Code Review Items

The following table contains a list of items to look out when performing code reviews. It is not an exhaustive list, feel free to add to the list as you see items during reviews.

 

  Requirements
R1 Reread the ticket/requirements so you know if the changes are appropriate for the issue in question
   
  Data Model
DM1 Each entity and attribute has been documented correctly, including business examples.
DM2 DB constraints should be in place for all entities/attributes, e.g. foreign keys, indices, unique indices for soft keys, etc...
DM3 Optimistic locked should be enabled, if it's not required a technical reason must be provided in the entity documentation.
DM4 All entity operations must have a description, except for standard operations (read, insert, update)
DM5 NULLS_OK must be set for each attribute, the default value should not be used (forces developer to think about value)
DM6 Physical deletes should not be allowed unless valid reasons are provided
DM7 When using localizable text, nulls should NOT be allowed for the foreign key to the localizable text record, also a foreign key should exist to enforce data integrity. The reason the ID must not be null is because the application always expects a value to be present even if the text field is optional.
DM8 Check text fields on an entity (e.g. name or description fields) use LocalizableTextIDs and not Strings
   
  DMX
DMX1 Only ship data that is necessary for the application to work - Any data that is not required for the application to function correctly or used for demos should not be shipped. Test/demo data should be separate to initial data and should not be shipped. (FIXME AB Might require rewrite for TI)
DMX2 Ensure data is in under the correct folder structure (initial, demo, test) (FIXME AB Probably not accurate for TI)
   
  Codetables
CD1 Name of code table file should correspond to the name of the code table
CD2 Appropriate package name has been assigned to the code table file (don't use 'curam.codetable' for project specific codetables). See tech note on packages.
CD3 Display name value has been specified and makes sense
CD4 Correct range has been used for code values (id allocation)
   
  Messages
MS1 Correct name format has been assigned to each message
MS2 Appropriate package name has been assigned to the message file (don't use 'curam.message' for project specific messages). See tech note on packages
MS3 Ensure placeholders are correctly used
MS4 Ensure message name and description provide enough context for a translator to be able to translate the message appropriately. Additional comments may be required and should be added above the description.
   
  Properties
PR1 Sufficient comments and doc comments have been provided for each property
PR2 Display name is appropriate
PR3 Use of application property is appropriate
   
  Model
M1 Facade classes and operations have appropriate documentation including javadoc
M2 Access levels have been set correctly (no source & documentation & access rules)
M3 Each entity and fa√ßade layer class should be placed in its own model package within the appropriate area of the model.
   
  Source
S1 Package and/or access levels are in place (no source & documentation & access rules)
S2 Visibility of classes/interfaces is correct. Only make public what is absolutely necessary, the standard should be to package protect classes/interfaces.
S3 PI Implementation visibility guidelines - All PI implementation classes (including DAO implementations) should be package protected. There should be no private operations within the implementation classes all private operations should be changed to default. This allows customers to extend our implementations (as a last resort) and reduces the surface area of customizations that customers may need to make. The constructor on the DAO implementation should be default, no need for a constructor on the interface implementation class.
S4 Appropriate PI events have been raised for significant operations
S5 Classes and operations have proper javadoc that describes in detail what the class or operation does. All parameters, returns and exceptions should be documented correctly.
S6 DAO implementations should be singletons
S7 No warnings in the code - both checkstyle and eclipse warnings should not be present in code
S8 IDs should not be passed around, instead the entity interface objects should be used
S9 Fa√ßade layer implementation classes should be declared abstract so that they can't be created using new, creating an instance of a fa√ßade layer class using new will cause logging not to work for that class. The fa√ßade layer implementation classes should also have a protected constructor which allows them to be sub-classed. All methods that are not modeled should be protected so that they can be overridden by sub-classes.
S10 No empty mandatoryValidations(), crossFieldValidations(), crossEntityValidations() allowed without a comment explaining why validations are not required.
S11 No empty catch clauses, unless documented why this is acceptable.
   
  Tests
T1 Test coverage should be above 80% for the classes/packages being reviewed, however more importantly ensure that all key paths and scenarios are tested correctly
T2 Ensure unit tests use test data and don't insert data in order to test view/update functionality
T3 Unit tests should be independent -
  • Tests should test only one thing at a time. Multiple assertions are okay as long as they are all testing one feature/behavior. When a test fails, it should pinpoint the location of the problem.
  • Tests should not rely on each other - Isolated. No assumptions about order of test execution.
T4 Don't assert the number of items in a list, this will cause breaks when new test data is added, instead check the size of the list before and check that it is has changed when you add/remove items.
T5 The test class should be in the same package as the impl class it is testing (i.e do not use curam.foo.test, use same package structure as impl class.) This allows easy testing of default operations and prevents duplication of test classes.
T6 Failure message should be provided with asserts. This helps people understand what we're testing.
T7 Test methods should have javadoc to explain what exactly the method is testing.
   
  Screens
UIM1 All overridden client screens should be placed in an overridden folder following the same structure as the component that they were overridden from
UIM2 Each field should have a help value
UIM3 No wrapping of RPX text at the minimum supported resolution (1024 x 768)
