Steps to submit a ticket

 

 

The following checkpoints must be met before workitem submission:

  1. Automated Testing: (TDD-style is recommended where applicable)

    All CRs that change java, javascript and XSL files should implement automated tests that cover the changes. 
    All automated tests should be run locally on the developer's machine before submitting:
          1. Java  

            All java code needs to be covered by JUnit test. JUnit is a unit testing framework for Java programming language. JUnit has been important in the development of test-driven development. 
   
      2. JavaScript

             Javascript code needs to be covered by unit testing. Intern framework is used to test the Javascript code.

             The intern is a complete framework for testing Web sites and applications. Itâ€™s built around standards like WebDriver and was designed from the ground up to be the most powerful, flexible, and reliable JavaScript testing system in the world.

  •  Running Intern within CDEj:  link
  • JavaScript unit testing and coverage: link

  1. Builds for smoked test and policing code:

    Run the following commands in cmd window:

- Client and Sever checks:

     1. ./appbuild policing

     2. ./appbuild  checkstyle
     3.  ./appbuild findbugs

- Small builds:

       1. ./appbuild  policing.client
       2. ./appbuild  policing.server
       3. ./appbuild  client.checkstyle
       4. ./appbuild  server.checkstyle
       5. /appbuild  client.findbugs
       6. ./appbuild  server.findbugs

 

 

  •  

  1. Comments and Modification history:


                  - All code changes should be appropriately commented.
                  - Modification history information should be added to any changed file that has a mod history section at the top. The ticket number and the developer need to be written as part of the comment.
                  - The copyright needs to be changed with the actual year.


    Modification history Example: 24-Apr-2010  BD  [RTC-000000] Re-enabling this test as an issue with mandatory labels has been fixed.
     

  1. Release Notes:  

    1. Internal Release notes:
      Small and succinct description about the impact for out of the box.
       

    2. External release notes:
  • There are some types of tickets that are required to write release notes. 
  • When it has:
    • APAR or PMR
    • An impact for the customer. CLE related with it
    • Enhancement
  • Following some tips that talk when it is required. Link wiki
  • Release notes attachaments. link
  1.  Manual Test Plan:

             - Execute the test for all the browser support in the application. 

             - The application support tablet in the 7.0.1.0. The application needs to be tested in all the tablet supported. 

Almost all the test needs to be automatic. In the case that the scenario should not be covered by the automatic test, the test plan should be written covering all use cases from related design page (standard users, JAWS users). The link to this test plan should be included in the Testing Information (in Testing tab)in the RTC ticket.
Tips:

  1. Be clear,

  2. Small steps,

  3. The QuickTime player app should be used to record the steps

 

  1. Finally..
     
    1. After all, tests are run and successful, RTC ticket can be submitted.  
      See more here on the topic Change set and Check in.
  1. Find the reviewer.
    After you agree on the reviewer, Go back to RTC and modify the owner of your ticket.
     
  2. After the ticket has been approved by reviewer it has to pass Smoke test and CI Build, at those two stages system will automatically notify you of the success or failure. If any of these will fail please seek advice from the reviewer or other team members.
