External Portals

If we are providing properties to control the behavior of certain features and these will be factored towards internal (caseworker) and external (citizen facing) applications, then we need to know which "infrastructure" is rendering these "external portals". There is no issue if they all use the Universal Access external container (as the external properties will all pertain to that) but if we have a mixture of CDEJ and UA infrastructures rendering "external" portals, then we do have an issue!

SamplePublicAccess Portal

- This is a demonstration of how you would provide a public portal for an entity to register and then login and see some of their details.

- CPM use a very similar (if not exact copy) model for the Providers. They can use the external portal to register, it's less secure and allows you to receive a registration email and then log in the the 'non-external' portal. My understanding is that these are provided primarily for third party's, e.g. an Employer or Service Provider, as opposed to 'the public' which is what UA is providing.
- As it is sample it is not something we would expect customers to use out of the box. We expect customers to take responsibility for building out their own version based on this example. For example they may want to customise the authentication part to be more sophisticated, and then customise the services that are provided, including authorization to resources.
