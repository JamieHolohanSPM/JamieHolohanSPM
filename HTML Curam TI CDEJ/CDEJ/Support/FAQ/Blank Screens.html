Blank and Frozen Screens

The page covers all support cases related to blank and frozen screens that have been countered. A blank screen is as described - the user gets a white screen in the browser with no content displayed. A frozen screen is similar, but may not always have a blank screen and may have the symptom of the browser not responding to input.

In general, the most common cause of frozen and blank screens, especially intermittent behaviour, is generally related to performance problems and potentially the number of connections open within the Browser.

Common Must Gather Information

  • Browser Version, including compatability mode settings

Can you please confirm the 'Browser Mode' and 'Document Mode' on the users machines encountering the issue. This is done by pressing F12 in the browser window and taking note of the browser and document modes listed.

Is this issue reproducible on Chrome?

  • Usage scenarios

Any usage scenarios that can be provided will help narrow down the cause of the issue.

  • Server response times

Any potential performance issues with the server?
Are the users locale or remote? Is there potential network issues with response times?

  • HTML Source for blank page

To get the HTML source in IE, right click and choose the View Source option.
Are there are any JavaScript errors in the browser?
Please provide screenshots and details if there are.

  • How do users recover from this issue?

Does the browser crash and need to be restarted?
Do they just refresh the page and start again?

  • Browser Javascript Logging

Client and browser tier debugging can be enabled using the following settings

TraceOn=true
JavaScriptTraceOn=true

These should be set in the CDEJResources.properties file, located in the curam/omega3/i18n directory.

At development time, this file can be added to your webclient project in a custom component, before building the application.

At runtime, a copy of the file can be found in properties.jar located in the WEB-INF\lib directory of the ClientModule.war file (within the Curam EAR file).

For example, in WebSphere, locate the client WEB-INF\lib directory installed in WebSphere:

<WAS_HOME>profiles\<profile>\installedApps\<node>\Curam.ear\ClientModule.war\WEB-INF\lib

Extract a copy of CDEJResource.properties from the properties.jar located in this directory. The file should have the package curam\omega3\i18n.

Place this copy in the <WAS_HOME>profiles\<profile>\installedApps\<node>\Curam.ear\ClientModule.war\WEB-INF\classes\curam\omega3\i18n directory and modify it to set the above properties as required.

Restart the server

Both options have performance implications, so may not be possible to set in a production system.

  • Enable browser JavaScript Tracing locally

It is possible to enable the browser javascript tracing locally for a user using a web debugging proxy tool.

There are a number of these online that can be used and we would recommend the Fiddler2 Web Debugger tool (http://fiddler2.com/).

Using the web debugging proxy tool, you should set the DEBUG javascript variable to true (default false). Then, using IE Developer Tools (access by pressing F12 in IE),view the Script Console view, copy/save the resulting debug output and send it in the PMR.

Note: IE Developer Tools must be open before reproducing the issue to catch the output and the output cannot be caught if the browser itself crashes as part of the issue.

If using Fiddler the following approach can be used to modify the DEBUG variable:

Select Rules -> Customize Rules ... in the Fiddler menu

In the file that opens find the onBeforeResponse function and add the following line:

static function OnBeforeResponse(oSession: Session) {
  ......
  oSession.utilReplaceInResponse("var DEBUG=false;", "var DEBUG=true;");

}

Save the file and any subsequent accessing to the application, with Fiddler running, will result in debug output.

In addition to the console log from IE, the traffic captured by fiddler itself can be used for debugging. This will allow us to see what network traffic is happening and if connections are being lost.
Note: You need to unencrypt the HTTPS content in fiddler. This can be done using the Tools -> Fiddler Options menu. Select the HTTPS tab and tick the decrypt HTTPS traffic.

Blank Screens on Modal in IEG

Symptoms

The following javascript error accompanies such a blank screen, seen when running an IEG script in a modal in IE9 running IE8 mode on Curam 6.0.4.4:

"Member not found." in dojo.js, line 32, character 44086

Note: If there is no javascript error with the issue, then the problem could be the lack of removal of the loading mask. If a HTTP error occurs when IEG script is running in a modal, the loading mask is not removed. This applies to 6.0.4.x and is covered by TEC-16158

Cause

This was an issue in IE9 using IE8 mode where the mode was being set manually by the developer in the browser. Where the mode is forced by the HTTP Server or a configuration property, this is not an issue.

Resolution

We now believe that the mechanism being used by the customer (in these environments where the issue is being reported) to instruct IE9 to use IE8 mode may not be correct and could be causing the issue.

Can we ask the customer if they are using the developer tools in the browser for achieving this (e.g. pressing F12 and setting this manually)? This is not an approach they could use in a production environment or one Cúram would support - the customer would have to send down the instruction from the server to the browser that would do this automatically. This method of changing the IE mode will not work with IEG.

=================================
Instructing IE9 to use IE8 Mode

There are two supported methods for the customer to instruct IE9 to use IE8 mode.

1. Use a configuration property in ApplicationConfiguration.properties
2. Use a HTTP Server to send this instruction to the browser.

The first method is the easiest one for the customer to use and the one we would recommend. In order to do this they need to set the "user.agent.compatibility" property in "ApplicationConfiguration.properties" to be the value "IE8".

Blank/Frozen Screens (MCSS)

Potential Causes

The most common cause of frozen screens, especially intermittent behaviour, is generally related to performance problems and potentially the number of connections open.

Where the advisor panel is in use, The advisor panel has a feature that polls via javascript every 2 minutes for updated information.

If the response times from the server are slow, either due to network or server issues, then this can cause too many connections to be opened.

There is a maximum number of connections that can be opened concurrently by browsers at any one time. The browser will stop responding to click requests at this point and appear frozen.

For IE 8 and Chrome a maximum of 6 connections is supported.

In addition, the evidence flow widget uses a similar polling approach for updating data. If this is also displayed in an open tab, it could be increasing the number of connections being opened.
 
One option is to disable the polling of these features by increasing the poll interval to a value of days or hours. This can be done by setting the following properties via the administration screens:

curam.evidenceflow.polling.interval
curam.advisor.advicerefreshinterval

Obviously doing this will mean that these widgets are not updated with relevant information without manually refreshing them.

Resolution

Ongoing - but currently appears to be performance problems on their end. They turned off the extra polling, including some added by their own custom code just to reduce the impact of this.

Connection leak in IE9 (DWP)

Information

Customer encountering blank screens and through their investigation identified the following reproducible scenario:

1) Log in as admin
2) Open a complex page such as "My Organisation"
3) Reload a sub-tab as fast as possible by clicking very rapidly over the Curam Refresh button
 
In a command window, watch netstat -an and search for connections to the Curam server. After multiple reloads, we get a number of connections stuck in CLOSE_WAIT or ESTABLISHED state. These linger, whereas normally connections in CLOSE_WAIT or ESTABLISHED state disappear in a few seconds.
 
This is enough to demonstrate the fault. But if you can manage to repeat it until 6 connections are stuck, this is the default maximum IE9 will open to a single website. Future requests to Curam result in white screens.
 
Increasing the max AJAX HTTP1.0 and HTTP1.1 connections allowed in Computer Configuration>Administrative Templates>Windows Components>Internet Explorer>Security Features>AJAX group policy can delay the white screen behaviour. But this affects all web sites, not just Curam.

Response

In response to the CLOSE_WAIT and ESTABLISHED states seen:

Connections are stuck in CLOSE_WAIT until the application (in this case IE9) closes the connection (socket.close()). Only then will the OS move the connection to the next state towards closing. If connections are not closing down correctly, the fault is potentially a problem in Internet Explorer.

For ESTABLISHED connections, these will remain open depending on the keep alive timeout settings the client and server have negotiated. The keep alive timeout is done at the HTTP level and is usually configured in the HTTP server. It usually has a value between 1 and 30.

Resolution

HP onsite have opened up a support case with Microsoft on this issue. Internal feedback from Gary Parker is that Microsoft have stated in a response - but not explicitly - that IE is not honoring the close connection request in some circumstances (what these are, they are not sure).  As such, connections are remaining open incorrectly which can't be used and it is possible to hit max connections causing the white screens. HP are asking explicitly if this is an IE bug.

HP have also done a number of changes, including increasing the size of the max connections in IE to 10 and that seems to have improved the situation. Gary is going to get a summary of the changes made by HP. TODO: Include Gary's update when available.

Proposal for Improvements

Most of the issues in this group are reported as Cúram issues even though they may not directly related to Cúram. The overarching problem is that under exceptional circumstances the Cúram application is not behaving in a predictable and helpful manner. Specifically when network reponse becomes slow or fails the application is frequently left in a bad state and errors are not reported in a useful way.

This proposal is aimed at improving the situation by providing a robust and consistent error handling mechanism for AJAX/iframe requests in the Cúram application.

Progress indicator

Provide a progress indicator that will kick in automatically if a request is taking longer that X amount of seconds. It should be a non-intrusive message in a place where it is sure to be noticed by the user. The message would disappear again when request completes successfully. This will help prevent users getting frustrated and breaking the UI by clicking ahead of the previous operation finishing.

Error reporting

In case of requests failures, we should display error message to the user.

Depending on the context and severity it should either be the same non-intrusive message as above (disappearing on the next user action in the UI), or a dialog window requiring confirmation. In any case it needs to inform the user of something going wrong and it must provide information about what the next steps are for the user.

More descriptive information for developers can be written to log.

Session timeout handling

Handle redirect to login screen, and/or redirects in general.

Retry policies

handle retry policy

Public API

The proposed functionality must be available as infrastructure service/API for use by application code, because there are a number of cases where they are using AJAX.
