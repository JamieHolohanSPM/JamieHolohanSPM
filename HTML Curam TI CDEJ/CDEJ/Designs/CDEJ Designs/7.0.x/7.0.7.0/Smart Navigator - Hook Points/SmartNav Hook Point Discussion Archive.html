Requirements

The original requirements provided are below but these were re-defined during analysis and those on the parent page represent the target the solution is aimed at.

 

  • Requirement 1: Display specific icons for a person in the Smart Navigator Person Search Results Dialog which convey various extra pieces of information about that person in the Smart Navigator. These may include:
    • Prospect Person
    • Special Caution ((when hovered over, displays a brief summary about the caution (sex, staff-security, public risk).
    • Duplicate
    • Secured
    • Must View Alert (displays a brief about list of notes, etc. (ex. IRD Change, Agent Update))
  • Requirement 2:Present a confirmation dialog or other mechanism to allow a caseworker acknowledge some key information before they proceed to use the rest of the application (normally from the person search results).
    • Actions must be allowed to be invoked from this dialog.
    • Once the dialog is dismissed, the rest of the application becomes active again.
  • Requirement 2 Expanded:
    • 1. Determine if extra information needs to be displayed before navigating to the target page
    • 2. Display extra information tailored to the client's situation (e.g. through a custom modal dialog)
    • 3. Get a forced acknowledgement that the user has read / understood the information.
    • 4. Invoke extra system processing if required
    • 5. Additionally, do not display this extra information when the same user is accessing the same person's record several times in the same session

 

Design Options

The following represent the design options as originally provided but have been moved here to avoid confusion. Design options given detailed consideration are on the parent page

 

Option 1 - Include the Icons which are displayed in the standard person home pages in the Smart Navigator Person Search Results

  • This would just take what we have with regards to the displaying of icons in the person home page and determine if we can display it in the Smart Navigator results based on the collateral we have there now.

 

 

Pros:

  • Would indicate to the user that there are special cautions for that person.
  • Make it more consistent with the standard person page.

Cons:

  • Would the real estate available for the Smart Navigator search results be sufficient to show these extra icons?
  • Would there be a possibility to hover over and see extra information for these icons (see requirements)?

Option 1 Analysis

  • Person search provided by default implementation curam.smartnavigator.target.impl.PersonSearchImpl
  • Implementation has a call to protected method getConcernRoleImageURL with signature
protected String getConcernRoleImageURL(final long concernRoleID,
    final boolean isProspectPerson, final boolean isRestrictedPerson)
      throws AppException, InformationalException
  • Providing a hook point here would allow complex custom logic in determining the icon.
  • ! Custom implementations must consider performance given this is called for every person result which could in theory be very large

The interface for the hook would have to include both image and alternate text for accessibility.

This additional field (alternate text) would also have to be modelled in struct for the results and the JS updated to use it.

 

Further analysis required around codeline

  • PersonSearch Interface and Default Implementation are both EXTERNAL but interface is not marked implementable so not sure if intention here is to allow custom binding?
  • Also the getConcernRoleImageURL is protected - not sure if that was intended as a way to allow extension but would recommend hook over extension for standard approach.

Requirements questions

1) Would we have to allow for multiple icons per result - if so the design of smart nav does not cater for that and would have to be revisited

2) Would we have to allow custom icons for each type of result? This should be possible in a similar way as described above for the Person search.

3) Would we have to allow for conveying additional information (beyond alttext). For example, if special caution would we have to have a way of displaying the caution detail (e.g. hover over) or would it be sufficient to have the icon with alttext 'Special Caution'. May need design input if the former.

Option 2 - Quick View With Actions From the Smart Navigator

  • Open a page/modal with additional information

  • Contents of this page/modal would be completely customizable For example – MSD could show on this page/modal that a client has a special caution

  • Can investigate if it is possible include actions on this Quick View – i.e. to ‘force acknowledge’ from here

 

 

Pros:

  • Users would have to focus/hover on each item in the list for the quick view to display, so this wouldn’t always appear perhaps?

  • Depends on how this was implemented?

  • Current use case is where the rest of the application is not available until the information is acknowledged, this would have to be catered for?

Cons:

  • Aligns with the Smart Nav concept is  not the definite endpoint

  • Actions should (such as “force acknowledge”) be on the definite endpoint ?
