Background

This issue has been initially raised by Ontario, and then by the Scottish Govt. Further investigation from SPM A11y SMEs highlighted the existence of the same issue also with other actionable elements (see table below in Requirements section).

Some discussion has been conducted with the wider IBM Accessibility community in order to validate the expected accessible behaviour. A group of engineers in PD looked at various fixes, that have been drafted then by Sharunas in this Design page.

PD is evaluating the possibility of communicating the accessible behaviour PD is going to implement to the customer, in order to check their reaction.

Requirements

Goal: Row specific actionable elements, within a list, should be labelled in a way they can be distinguished from each other when using a screen reader.
 

List of Requirements

  • Requirement 1: Row specific actionable elements, within a list, shall be labelled* including the specific row number, i.e. "[...] row 3 [...]".
    • Requirement 1.1: The label* associated with the Link shall be "<link text> row <row number>", e.g. "Select row 2" for the Select link beside Granpa Smith (see Link screenshot below)
    • Requirement 1.2: The label* associated with Radio button and Checkbox shall be "Select row <row number>".
    • Requirement 1.3: The label* associated with Toggle button shall be "Toggle row <row number>".
    • Requirement 1.4: The label* associated with Menu button shall be "List actions menu row <row number>".
  • Requirement 2: The row number in the label* associated with the actionable element shall be updated to reflect what the sighted user sees on the screen, when the impaired user performs one of the following actions: sorts the list, uses the pagination feature, changes the number of results in the list.
  • Requirement 3: The row number that has to be added into the label* shall keep into consideration the presence of the header or not, being in sync with the actual HTML rows into the table. For example, the row with Granpa Smith is the row 2, as we have the header that is row 1 (see Link screenshot below).

*label = text associated with the actionable element (e.g. through title attribute, alt attribute, actual label, etc.)

NFR Note                                     
Accessibility

The ability of the system to be used by people with disabilities. When new technologies are being used for the UI very careful consideration should be given to whether screen readers can be used with the technologies. This applies specifically to usage of JavaScript, Flex and SVG. The use of colors to provide functionality should be considered (and avoided), along with resolutions that will be used by vision impaired users.

Both desktop application and tablet devices should be considered in this effort.
Availability & Resilience The ability of the system to be fully or partly operational as and when required and to effectively handle failures that could affect system availability. This should be considered when working outside of the normal infrastructure and deployment. For example, the XML Server is not deployed in an application server and has special handling for fail-over and load balancing. Careful consideration should be taken before adding new components, e.g. EJB Beans, MDBs, to the infrastructure.
Development Resource The ability of the system to be designed, built, deployed and operated within known constraints of people, budget, time, etc. In general there is not much information required here, however where special skill sets or people with siloed knowledge are required to work on the change/enhancement, this should be detailed here.
Evolution The ability of the system to be flexible in the face of the inevitable change that all systems experience after deployment, balanced against the costs of providing such flexibility. Consideration should be given to future proofing the change, including considering how to upgrade to any changes or enhancements provided in the future. Specifically designs should accommodate elements such as localisation (text, dates, etc.), customization, configurability, and componentization. Providing these features in future releases, when not considered up front, has caused impacts for customers in the past and has prevented us from providing the features in some cases.
Internationalisation The ability of the system to be independent from any particular language, country or cultural group. Localisation and internationalisation are a large concern for Cúram. Support should always be provided to view information in multiple locales, based on the user and server locale, along with support for working in multiple time zones.
Location The ability of the system to overcome problems brought about by the absolute location of its elements and the distances between them. For the most part this is not a major concern, however we should never design based on the assumption that the customer will use a default configuration
Performance and Scalability The ability of the system to predictably execute within its mandated performance profile and to handle increased processing volumes. Performance and scalability are a concern and should always be considered during design. Where necessary, the Performance and Benchmarking teaming should be engaged early to help with determining impact of potential changes.
Regulation The ability of the system to conform to locale and international laws, quasi-legal regulations, company policies and other rules and standards. Regulation includes the usages of third party tools and the licensing issues associated with these. New jars should never be introduced without considering the legal implications and licensing of the jars. This includes usage of GPL licensed code. Also, usage of code freely provided on the web and not properly licensed should be avoided.
Security The ability of the system to reliably control, monitor and audit who can perform what actions on what resources and to detect and recover from failures in security mechanisms. This is a concern where new features are added that are not contained within the normal Cúram security.
Usability The ease with which people who interact with the system can work effectively. How the change will affect the user and how they use the application/feature should always be considered. Where necessary the business analysts and our user interface experts should be engaged early to ensure we meet usability requirements. e.g. printing.

 

Design Options

This is optional. If there are is more than one design option available and being discussed, then these should be outlined here.

Option 1. Pre-defined Readings

The technical overview of the changes to be made if we went for the accessibility readings based on the UIM pre-selection (specifying the fields used as the source of the readings) could be found on the ticket. This would allow for the greatest control over the readings contents and to avoid excessive and not informative texts being read out; on the other hand the extent of the changes to be made and the application impact would far exceed the initial scope of the task (reported as a Customer defect), and would also require significant effort to design, implement and test such a change. Therefore, this is the least preferable candidate from the technical perspective.

Pros:

  •  full control over the reading contents
  • reasonable length of the readings allowing for reliable identification of a record.
  • the readings are output when rendering the page, hence available to the screen reader in all situations

Cons:

  • deep infrastructural change required with the UIM extension
  • too much design, implementation,testing effort
  • application impact and big take-on

Option 2. Full list row contents as a reading

No selective announcements, full concatenated texts are read (each pre-pended with the headings)


This option takes into account that the DOM focus events are fired for the input controls and handled before the screen reader kicks in, therefore attaching onfocus handler which collects the current record values into the readable text would set up the proper reading.

 

Since the values of the records of the records are just available upon page rendering, the task is mostly a web-scrapping, where JavaScript scans a list with the controls to collect and set up the right information. The process based on the DOM querying is always inefficient, and the repetitive queries is a bad code practice; therefore an effort should be made to store the once collected values so that they can be re-used afterwards.


The entry point into the row can be made up by a positional or identified querying of the list DOM. On the other hand this would either increase the page footprint by adding identifiers (these are available for the input controls like radio buttons, but aren’t normally output for the ‘Select’ type links), or resort to the non performant and non stable positional or semi-positional querying based on the list table style names, table ID’s, etc.


Since the list output is controlled and set up at the JSP generator level where the list identifier is known (each of the lists regardless of them being paginated or not is assigned the on-the-fly generated ’pagination-id’, which is already used by JavaScript for pagination. Furthermore, the JSP generator resolves to the control fields (links, radio buttons, checkboxes) at the particular points, hence the scenario where the first column contains such an ‘active’ entry is known and can be marked specifically to add the information required just for the lists with such controls.

 

And while the lists in Curam are not separately rendered entities like widgets, the repetitive character of the information requires JSP iteration which is handled by the LoopTag. This tag does not have much attributes, just setting up and maintaining the shared loop index (record number) when rendering. At the same time, being a JSP tag and Java class it can easily intake the additional attributes and also be reached by the other tags, logically enclosed within it - by either using JSP ‘runtime traversing’ or, more efficiently by using the shared rendering environment known as JSP page context, where the shared variables could be set and maintained during the rendering time. This allows for information collection at the tag level and also delayed output thus allowing for right HTML-JavaScript statement order on a page, based on the right interaction with JSP generator.


For a list with controls we would need the list identifier (there could be multiple lists per single page), indication if this list contains any controls, the information of the list headers (this could be done later by JavaScript and yet, this would be completely excessive for the information known at the rendering time), and the onfocus event for each of such controls; this latest addition allows for avoiding the additional external DOM querying, since the current object is available as ‘this’ at the point of invocation, and therefore the DOM querying could either be replaced by the DOM traversing (using parent relationship), or shortened by querying the surrounding DOM for the closest (enclosing) row and thus getting the required entry point for building the record readout.

 

Technically the approach does not have much drawbacks, since the information is distributed more or less evenly and the JavaScript part is minimal; furthermore, caching the once collected record readout in the memory representation of a DOM element (control) allows for straight re-using it without reprocessing, and is also insensitive to the sorting and pagination (since the once set readout ‘moves along’ with the control). The accessibility and usability perspectives of the long readout with the substantial and non-substantial values dump in the reading is a different issue and belongs to the corresponding design.

Option 3. Row (record) number as a record identifier

In this scenario the list row order (numbers) rather than the record contents are announced as the selection base (e.g., "Select row 3").

Most of the principles as Option 2 describes are also applicable for this option, however there is also a complication: the row numbers which are to be announced cannot be relied upon as static - they are actually changed with every sorting or pagination operation. The information about the row it belongs to cannot be stored in the control memory, since it now can belong to a row with a different number. Web scrapping each time when a sorting operation occurs is counter-productive and would in any case require some additional JS caching and management on the list-table objects being affected; given tables are not 1:1 bound to the lists but used also for the general layout tasks (e.g.,  clusters resolve to tables as well), the subset which represents the lists on a page needs to be memorized.


The considerations like these lead to a fall-back approach with Option 2 modification so that the initial information on the lists is output on a page even though no header maps are needed; the information again should be distributed so that it is used at the earliest point where it becomes available, either being consumed or facilitating its detecting and consuming at the later stage.


Therefore, the list row should provide for storing the record number information and output the initial numbers for it. With HTML5 custom attributes this does not break the validity of XHTML and allows for relatively compact way of expressing this information as well as the slot for keeping the updates when these occur.
The HTML list rows are output from the generated JSP; the loop index maintained for every list also corresponds to the record number (+1 correction since loop is started with 0). Hence, the additional attribute “data-lix” (“line index”) is introduced in the page DOM and attached to the HTML table row element TR.
The list identifier at the same time allows for outputting named variables into the JavaScript domain, so that they cannot be applied to that particular list.


The controls on the list are collected by the loop tag and added to the map associated with the list via the named variable (expand on that). Each control outputs it’s identifier for the JavaScript.


As the JavaScript runs, those identifiers are converted into the DOM object references, from which the corresponding rows are queried. The row index migt change because of the sorting/paginating or any  other movement, however the row and the control bound will remain; therefore, the control object can store the associated row; same way, the arrays initially filled with ids can be replaced by the row references for the subsequent quck access. Thus the approach where the information is just collected once pre page life-cycle is used here as well.


Finally, the JavaScript which sets up the accessibility readings is run initially but also needs to be engaged by any operation which changes the current list order; currently this is the sorting operation and pagination operation, correspondingly the JavaScript in the sorting and pagination modules is amended to have the list reading updating call. However, in order to make this loosely coupled, an event-subscription approach is used, where the initial setup subscribes to the pre-defined event, which is published upon the sorting or pagination operations (like going to the next, previous page or changing the pagination size).
Just to note, the script which initializes the list readings run at the very end, and is therefore added to the general page scripts. There is no overhead if no list with controls data exists on a page; if there is the data, it is processed as described. As for the data itself, an important thing is to place the accumulated named arrays after the list HTML has been output. This can be and is achieved again by page context mediation; this shared rendering environment is capable of temporary storing of the accumulated information, which is just output on a page once requested; the request to output it is added via the JSP generator to match the ending of the list processing template where all the list HTML is 100% output already.

Table Support

 

  • As of v7.0.1 Curam will be certified for use on tablet devices. This introduces a new responsibility on Business Analysts, UX Designers and Developers to ensure that they do not change existing or introduce new features that break the Tablet support statement. For the most part this is limited to the production of custom widgets (non-standard UIM) and how they perform when using touch screen controls. Please note the following:
  • Tablet resolution is typically lower than the minimum recommended resolution of 1366x768. This means that the rules around wrapping of text have been relaxed for tablet, but teams should make a reasonable effort to ensure that any wrapping is not excessive and does not affect the ability to understand and use the application.
  • Where custom features are introduced, teams must test using a iOS and Android device to ensure that the feature is compatible with touch screen controls.
  • Where custom features are introduced, they must meet accessibility requirements for mobile devices.

Implementation Notes

The actual implementation (details to be added later when I have a minute) is a mixture of the output by the JSP generator and web scraping to mark the list objects with the dynamically set accessibility readings (links, toggle controls, list action menus) so that they can be found on a page and added to the rows they belong to.

The markup is in the HTML attribute form; the links are marked with the 'data-control' attribute, the toggle controls have a separate 'data-toggler' attribute and it is the 'data-notify' for the list action menus.

All attribute values are set to the current list identifier either at build time (for the toggle controls and list action menus) or at runtime (for the links). The later ones source the list identifier from the page context available at the page JSP generation time and placed there by the LoopTag when processing the list markup.

 

The lists on a page are initialized upon the page load, and only if there is a list on a page with any of the mentioned controls marked in the corresponding way; the page is scraped by the JavaScript to find and properly the properly marked controls.

 

Currently the  links covered by the fix are the ones resulting to the <curam:link> tag in the generated JSP; they have the 'data-control' attribute properly added and set. As opposed to that, the links output by the <curam:popupReturn> tag are unknown to the accessibility reading update system. In order to cover them, they should have the 'data-control' attribute with the current list identifier set on them, this will result in

  • adding the arial-label to those links
  • updating the aria-label reading when the list is sorted or paginated.

The popup return tag is being output by the JSP generator (gen-jsp.xsl) and hence the the attribute could be added there - checking for the LIST ancestor and, if the control belongs to a list, adding the instruction to get the value from the page context, something along the lines:

   <curam:popupReturn data-toggler="${{currentListId}}">

This, however, requires further processing within the JSP generator so that the attribute is just added if the popupReturn tag belongs to a list (which might require more complicated jsp:attribute based set up), modification of the tag description in curam.tld to accept the optional new attribute, and also the modification in the PopupReturnTag code to use the new attribute in the output.

A simpler, and hence recommended approach would be to stick to the Java code changes only - mostly in the PopupReturnTag. The required changes are drafted in the comments of  Defect 248363, so just repeating them here with further explanation:

The PopupReturnTag output is done by the doAfterBody method. Therefore that should be updated in the following fashion:

  1.   Within the if body != null { clause, first thing is to add the following variable initialization:

    final String listId = (String) pageContext.getAttribute(LoopTag.LIST_ID);

      The curam.omega3.taglib.flow.LoopTag should be imported along.

  1.    The first occurrence of the body.getEnclosingWriter().print(... statement is to be split so that there is a place for another attribute (after the href=\"#\") :
    body.getEnclosingWriter().print("<a class=\"ac\" href=\"#\"");
  2.   The new conditional output should be introduced like:
    if (listId != null && !listId.equal("")) {
      body.getEnclosingWriter().print(" data-control=\"" + listId + "\"");
      pageContext.setAttribute(LoopTag.HAS_CONTROLS, "true");
    }
  3. Another body.getEnclosingWriter().print(... should be added to output the remaining code:
    body.getEnclosingWriter().print(" onclick=\"" + mappings  + "setParentFocus(arguments[0]);return false;\">");

The rest of the code remains untouched.
This should be sufficient to make the list control managing scripts to be aware of the additional control to set the accessibility readings to.
