Conditional Codetable Hierarchy

Design Page for Conditional Codetable Hierarchy

Background

At present, the codetable hierarchy is static by design, i.e. it must be specified in a single CTX file, and cannot reference other codetables. An enhancement to the supervisor workspace depends on a hierarchy of case categories and case types, however, due to the existing design of the infrastructure, a series of search pop-ups are used to mimic the hierarchy.

Requirements

  • Task 107360 - Conditional dropdown for the codetable files which are coming from different components

Design Options

  • Option 1

    • Add markup to existing UIM elements to specify search endpoints (using existing codetables) a filter class, which follows a contract set out by the infrastructure
      • The filter class can be invoked for regular codetables to allow display of a subset of values, or for codetable hierarchies, to achieve the same.
    • Application developers are responsible for filtering the results to meet their business requirements

  • Option 2

    • Add a new UIM element widget to handle the dynamic filtered codetable hierarchies
      • Also requires a filter class to be provided by the application developers for the filtering functionality.

  • Option 3

    • Add markup to existing Codetables to establish a relationship
    • REJECTED
      • This would be identical to how codetable hierarchy currently works, and not dynamic as required

  • Option 4

    • Create a wrapper codetable hierarchy for existing codetables
    • REJECTED
      • This would be identical to how codetable hierarchy currently works, and not dynamic as required
      • This would require model changes, including a new Domain Definition, and updates to Struts.

Resilience


This is mandatory. Some thought should be given to how resilient the new feature is.

  •     What happens if something expected/unexpected goes wrong?
  •     What happens to the system?
  •     Will the user be able to take corrective action in the system without the need to resort to application logs/SQL searching/data manipulation outside of the system.
  •     Will the data be left in a consistent state within the system after the exception is thrown?
  •     Will there be messages given to the user? What will these messages be?
  •     Will there something written to the logs? What will this be?
  •     Does the feature require some of the exception data to be captured and an admin function provided to allow the situation to be re-mediated?
