Session Timeout Warning

 

Design Page for Introduce the Session Timeout Warning 

Background

 

There has been confusion with the end users to understand when a server interaction is made, and eventually losing data on some of the screens. As of now, Curam application does not let the user know when the session would be timed out, and when was the last interaction made to the server. 

Not every click on the application rightly so does not make server calls, and there have been numerous instances user assumes they are active in the application, but when the next server call is made, the user is forced to re-authenticate to the application. There are some cases where the user inserts data, for example, a contact log for more than couple of hours, and user assumed that keying in the comments field is being active, and with these confusions, after the session timed out user is forced to re-authenticate and eventually losing the data which has been extremely frustrating to the end user.



Requirements

EPIC -107499 - Introduce a session timer into the application to let users know when the session will time out.

  1. This notification should take the form of a modal window being presented to the user with the following specifications (as default)
     
  2. If a user does not interact with the server for a specified period of time in the application, they should be notified by the application that their session is about to timeout.
     
  3. A timer countdown which details how much longer the user has before the session does time out (and list the default for this). However, there is also an extra buffer size that could potentially be configured on the application server for the timeout which should also be taken into account.
     
  4. The message to explain to the user what is happening.
     
  5. A button to allow the user to restart the session and continue their work.
    1. The modal dialog is closed;
    2. The system resets back the session timeout;
    3. The user can interact with the application.
       
  6. A button to allow the user to not restart the session.
    1. The modal dialog is closed;
    2. The application is redirected to the logon page defined by the property.
       
  7. When the session times out without any user interaction:
    1. The modal dialog is automatically closed;
    2. The application is redirected to the logon page defined by the System admin or the default logon page with a message informing the user that the session has timed out.

Resilience

  •  What happens if the application can't read the session expiry cookie or properties?
    • The application is not able to execute this feature
  • What happens if the default properties are configured in the CDEJResources.properties and ApplicationConfiguration.properties but the feature is not enabled by the System Admin with the property curam.environment.enable.timeout.warning.modal  or curam.environment.internal.enable.timeout.warning.modal?
    • The application is not able to execute this feature
  •   What happens if the configuration properties are not able to share between both applications(external and internal)?
    • A new solution needs to be thought.
  •   What will happen if the values of cookie are changed maliciously?
    • The session timeout will not appear as while the cookie is invalid. We could have an endless loop in this case. The session timeout warning will never appear in this case.
  •   What will happen if the session is reset when any modal dialog or the application does some redirect for a second page?
    • The Filter will reset the cookie with the new values.
  •   What will happen if the user doesn't like this feature?
    • The user needs to talk to the System Admin to switch off this feature.
  •   What will happen if the System admin decides to switch on this feature?
    • The System admin needs to follow the steps to turn on this feature.
  • What will happen if the System Admin would like to change the default configuration properties per application or globally.
    • The System admin needs to change *.app XML file with the element timeout-warning defined above.

Implementation Notes

 

 

              The session timeout warning API is started in all the applications that extends:

               1) The appController.jspx (internal application) or

               2)ExternalApplicationBannerRenderer - external application for Authenticated user or

               3) resolve.jspx and player.jspx - external application for Not authenticated user.
 

Configuration Properties

             -  External Application

  1.  The ExternalApplicationBannerRenderer Class(banner component) calls the JSPUtil to recover the session timeout warning string to insert the javascript in all the pages.
  2. The curam.util.client.jsp.JspUtil.getSessionTimeoutWarningInitializationJavaScript(appID, sContext) return the javascript responsible to start the session timeout javascript check;
  3. There is a javascript called SessionTimeout.js that is responsible for all the logic of this feature.
  4. The LandingPageRenderer class(home page renderer) calls the JSPUtil to print an expired message on the page.

             -  Internal Application

  1.  The appController.jspx call the JSPUtil to recover the session timeout warning string to start to check the session timeout warning.
  2.  There is a javascript called SessionTimeout.js that is responsible for all the logic of this feature.

    -  Manage properties between Internal and External application

  1. The JSPUtil class create a string with the session timeout warning javascript to be inserted in both applications.
  2. The curam.util.client.jsp.JspUtil.getSessionTimeoutWarningInitializationJavaScript(appID, sContext) return the javascript responsible to start the session timeout javascript check;
  3. The properties is recovered calling  the class TabLayoutResolver.java using the path parameter: "/config/tablayout/timeout-warning[app-id][app-container]".
  4. The class TabLayoutResolver is used to get the properties. 
    1. The Class TabLayoutResolver has all the logic to return the JSON based if the application has the specific configuration or the global configuration needs to be returned.
    2. The step timeout-warning is responsible for getting all the properties according to the priority.
