Progress Widget

Background

The goal of the Progress Widget is to inform the user that an action that is happening in the server causing the screen that is current being displayed to be blocked, preventing any kind of user interaction.

 

Problem Description

User receives no visual feedback they have completed a UI action successfully and are awaiting its completion. In most cases a blank screen is displayed as the request is being processed. The user is left unsure if they somehow clicked the wrong link, caused an error or if they need to refresh/close the page.

 

Requirements

Design Requirements

  • Requirement 1 -> Discuss and define the default background color to be used for the overlay when the application is loading content / busy.
    • White was selected as the default color.
  • Requirement 2 -> Discuss and define a default spinner image (gif) to be used.
    • The IBM Northstar Loading Spinner was selected.
    • https://www.ibm.com/standards/web/design/loading-spinner/
  • Requirement 3 -> Discuss and define a default threshold value for when the component will start to be displayed.
    • 2000 milliseconds (2 seconds) was selected.

 

Functional Requirements

Decided on as part of the investigation and implementation of this Progress Widget.

  • Spinner shall have a turn on/off mechanism which shall enable or disable the Progress Widget across the application. The Progress Widget shall not be configurable on a page-by-page basis.
    • Configurable under the property "curam.progress.widget.enabled" define in ApplicationConfiguration.properties.
  • The Progress Widget panel shall block the whole application once displayed, avoiding to the possibility of multiple spinners appearing.
    • The overlay provided by the Dojo Standby widget will be used.
  • Progress Widget spinner shall be displayed within the element/frame which triggered the application to load.
  • The Progress Widget shall have a timeout triggered when the process which caused the Progress Widget to display fails. This ensures the Progress Widget will be removed and stop blocking the application.

Design (Implementation)

Create a new module in TI that will act as a wrapper for the actual Standby component, this module would have few overloaded methods depending on which kind of customization the application team want to do, but at the lower layer we will have default values defined for when no values come from the application team.

 

The dojo Standby widget allows the following customization's that are used in ProgressSpinner.js: 

Parameter/Variable

Description

target

  The target DOMNode id, DOMNode reference, or Widget ID to overlay when show() is called.

image

  The URL of the image to center in the blocking div overlay

imageText

  The ALT tag text for the centered image.

color

The background color of the translucent DIV overlay                                                                                                                                  
text                           Text/HTML to display in the center of the overlay This is used if image center is disabled.                                                                                                                                                                         
SHOW_SPINNER    

The SHOW_SPINNER variable was added to the ProgressSpinner.js to                                                                                                      

dismiss the progress spinner on the Frequency Editor when incorrect/invalid values are entered                                                                        

Methods which were created (ProgressSpinner.js):

  • init()
    • Initialize progress topics.
  • addClassAppendChild (pageSpinner);
    • Adds the domNode that will hold the image and will have the background color to the body of the document.
  • _getSpinner (target);
    •  Instantiates the Standby dojox component and populate the attributes that we need. It will also call the startup() method of the pageSpinner reference that is needed by the Standby component.
  • show (pageSpinner, threshold)
    • Displays the spinner (Standby component) to the user after the Threshold amount.
  • dismissSpinner: function(context)
    • Removes the spinner from the DOM hierarchy.

Resilience


When using the feature with a custom Image and ImageText, a path must be informed so the application can find the image. If that path is incorrect an error icon will be displayed but the application will continue working as expected. A fallback mechanism for when an image path is incorrect was considered but still need to have a better understanding of the requirements so it will be developed and implemented in a different task.

Documentation/How To Use Feature

 

This enhancement will be used every time a submit button is clicked, the feature has 2 working modes, first one will display a spinner image with some background-color overlaying the application screen and the second one displays a defined text with the overlay layer so that the user can understand that some processing is happening in background of the application and that they need to wait it to complete. The advantages of using this feature is that the user will no longer think that the application is just hanging there and instead they will realize that something is really happening and that they need to wait until that processing is done. The feature is highly customizable, that means that the application team can define the background-color, image, text to be used.
