HOW TO TAKE ON THE NEW "JDE ONE DELIVERABLE"

 

1. Add JDEVersion.txt to your ExternalVersions folder, containing the following: (NB: The line beginning with "zip.version=" will  have to be modified to match the version of the zip you are taking on)

zip.name=CuramJDE
zip.version=_7_0_0_0_test1
zip.network.location=ProductDevelopment/DevReleases/SDEJ_CDEJ_release
zip.unpack.location=${sysenv.CURAM_DIR}/JDE
zip.post.unzip.file=${zip.unpack.location}/jde_post_unzip.xml
zip.post.unzip.target=main

 

2. Remove the following files from ExternalVersions folder if present:

    - CdevenvVersion.txt

    - CuramRSMPluginVersion.txt

    - EnvVarsVersion.txt

    - GSSVersion.txt

    - GlbnToolsVersion.txt

    - PolicingVersion.txt

    - SdevenvVersion.txt

    - ToolsVersion.txt

 

3. Bootstrap, build and test everything as you would normally.

4. Commit JDEVersion.txt plus the deletion of the other text files to source control.

5. That's it. You are done. The rest of this page is the design page.

 

 

Design Page for JDE One Deliverable

Background

Currently we release 9 zips with a JDE release: SDEJ, CDEJ, EnvVars, GlbnTools, InternalPlugins, Policing, RSM, UDP and UIToolkit.

Although it is not strictly necessary for consumers of these zips to keep everything in sync it would be recommended.

There is also takeon effort required to keep those zips in sync.

A better idea would be to release a single zip containing all of the above tools so that there is just a single takeon step required.

 

Requirements

1. Convert nine zips into one zip

2. Have a single takeon step for consumers

3. Tools are unpacked in the same way as they were before

 

Ticket: RTC 149493 - JDE One Deliverable

Design

  1. Modify pack.xml to generate the new zip containing all nine folders
  2. Include a jde_post_unzip.xml file that will move the tools to the appropriate locations, run all post-unzip targets, check guard files, etc.
  3. Higher offsets need to add a JDEVersion.txt file and delete the old version text files

Notes

  • Why is ADBuildTools still separate?
    • ADBuildTools is bootstrapped using a different process than the other JDE components. Bootstrap uses an old version of ADBuildTools unzipped on tiny to pull down the latest ADBuildTools. Bootstrap then uses the new ADBuildTools to pull down everything else. So the new ADBuildTools is required in order to download the JDE zip. Therefore we cannot include it in the JDE zip!
