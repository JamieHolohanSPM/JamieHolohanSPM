Notes Printing Changes

Design Page for 251487 - The Internet Explorer 11 browser may quit unexpectedly when attempting to print a large number of Notes with different styles and formats

Background

When attempting to print a large number of Notes containing different styles and formats from the Client Contact tab on a Person page, it may cause the browser application to quit unexpectedly. This issue only occurs with Internet Explorer 11. This mainly occurs when attempting to print a large number of Notes with different styles and formats.

 

Requirements

Note, the issue being reported here only occurs in Internet Explorer. It is our view that we should make the printing process for a Notes page as efficient as possible from our side. If the customer attempts to print Notes which equate to a large number of pages with complex Notes (Maybe 150+) their browser may still quit unexpectedly, this is considered to be a limitation of the Internet Explorer browser rather than something we can really try to solve.

  • Requirement 1: On a Notes page with context panel print enabled, clicking the page level print icon shall launch 2 print dialogs.
    • Print dialog to print the context panel.
    • Print dialog to print Notes as the appear in the main content area.
    • Both shall print successfully on all supported browsers.
  • Requirement 2: On a Notes page with context panel print disabled, clicking the page level print icon shall launch 1 print dialog.
    • Print dialog to print Notes as the appear in the main content area.
    • Both shall print successfully on all supported browsers.

 

Design Options

Option 1

  • Specific to Notes pages only.
  • Upon expanding a Note in the main content area.
    • Overwrite the page level print icon to call a new function CEFUtils._overwritePrintButtonToPrintNotesPage().
  • Upon clicking the page level print icon on a Notes page which has been overwritten.
    • Create a new hidden iFrame off screen which is not visible to the user or accessible to an accessibility user. This iFrame should be a lightweight version of the main content area containing Notes. Rather than having multiple nested iFrames, hidden elements, defining CSS files, defining javascript files etc. this new iFrame should remove that overhead and essentially be made up of tables, divs and some styling.
    • Populate the iFrame with Note content exactly how they are displayed on the page.
      • Any non-expanded Notes should just have their header added to the iFrame.
      • Any expanded Notes should have their header and content added to the iFrame.
    • Call the inbuilt browser print() function on this new iFrame instead of on the main content area iFrame.
  • Alternative: Originally the idea was to dynamically update and iFrame in the background as Notes are expanded/contracted so the iFrame is ready to print whenever the page level print icon is pressed. This became too complex when dealing with changes in sorting the list of Notes and pagination. Also the overhead of generating the iFrame upon clicking print does not have impact on performance.

Requirement Decisions

Requirement Decision Title Issue or Problem Statement Assumptions Motivation Alternatives Decision
Overwriting the page level Print icon If a user attempts to print a page with Notes without expanding a Note first the page level print icon will not have been overwritten yet. Printing 45 row headers using the old curam.util.printpage() function will not cause the browser to quit unexpectedly. It is better to keep logic for this fix outside of infrastructure. There is NoteHistoryRenderer.java  in CEFWidgets which is where this logic should live. Keeping code out of TI for this would have meant a more hacky solution. curam.util.printpage()  can print up to 45 Note header rows fine without crashing so only invoking the new print functionality from when the user expands a Note is fine.
Browsers Even though Internet Explorer is the only browser which quits unexpectedly should the new print functionality be called across browser. The new iFrame solution can work across each supported browser. It will be developed and tested to work on each supporting browser, or specific checks could be added in javascript to determine if the browser is Internet Explorer or not.   It was decided the new print functionality should be invoked on each supported broswer. This was decided for 2 reasons, consitency and because the looks of printed Notes in the new iFrame was deemed to be an improvement on how they look when printed from the main content area.
Styling Should elements in the new iFrame follow the styling of how they appear on the page.     It may be better to keep the content in the new iFrame as basic as possible to reduce overhead when printing. It was determined elements in the new iFrame should be styled as they appear on the page once copied to the new iFrame. Rather than importing our full style sheets, styling was added for each specific element using the "style" attribute to reduce overhead.

Design Resilience Measures

 

What happens if something expected/unexpected goes wrong? If Internet Explorer quits unexpectedly there is no way to to recover it apart from refreshing the page or restating the browser.

What happens to the system?

 N/A
Will the user be able to take corrective action in the system without the need to resort to application logs/SQL searching/data manipulation outside of the system? Yes
Will the data be left in a consistent state within the system after the exception is thrown? Yes
Will there be messages given to the user? What will these messages be? Messages from Windows indicating the IE has quit or become unresponsive.
Will there something written to the logs? What will this be? No
Does the feature require some of the exception data to be captured and an admin function provided to allow the situation to be re-mediated? No
