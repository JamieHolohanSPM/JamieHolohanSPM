Background

**Note**: From here on in, I use the terms focusable field/element or editable field/element. The difference between the two is, a focusable field is any field that the util.doSetFocus() function determines to be the focusField. An editable field, is a focusable field that requires Forms Mode to be enabled. But there are certain field types that require user interaction, but do not trigger Forms Mode. Examples of these are checkboxes and radio buttons, All editable fields are focusable fields. Not all focusable fields are editable fields. Below I've listed some examples of these:

Focusable field: Text input field, dropdown widget, a select, a radio button, a checkbox, an anchor element, a text area, a Browse button
Editable field: Text input field, dropdown widget, a select, a text area


A support ticket was raised by MCSS in relation to forms mode not being triggered for the first editable field for JAWS users in IE11 when a page has loaded. JAWS has the automatic forms mode setting switched on by default, so usually when an editable field receives focus (through tabbing/using JAWS virtual cursor) forms mode would be triggered, which would allow the user to interact with the field. For example, they may type into the field, or expand a drop down menu.
But in Curam, when the page loaded and the first focusable element was an editable field, forms mode was not being automatically triggered and therefore users were not able to interact with the field initially.
A fix for this was put into 7.0.5.0 as part of https://jazz031.hursley.ibm.com:9443/ccm/web/projects/SPM#action=com.ibm.team.workitem.viewWorkItem&id=225885.
The fix was to include a one second delay in setting the focus in IE11 when a page had loaded. This almost consistently (though not 100% of the time) triggered forms mode for JAWS users.

Since this fix was added to the codebase, a number of issues related to the change have been raised. For example:
https://jazz031.hursley.ibm.com:9443/ccm/web/projects/SPM#action=com.ibm.team.workitem.viewWorkItem&id=255274 was raised as if a user began typing in another field, the focus would jump back to the first focusable field after one second. This was addressed by detecting if the user had focused on another field, and if so, refrain from setting the focus on the default focusable field.
https://jazz031.hursley.ibm.com:9443/ccm/web/projects/L2%20Support#action=com.ibm.team.workitem.viewWorkItem&id=258299 and https://jazz031.hursley.ibm.com:9443/ccm/web/projects/L2%20Support#action=com.ibm.team.workitem.viewWorkItem&id=259828 were raised as if a user began to scroll immediately after the page loaded, when the first focusable element was focused, the page would jump back towards the top of the page. This was interrupting the user's scrolling and causing frustration for customers.

A TestFix (7.0.7.0_iFix5_TestFix_260301) was given to MSD to address the scrolling issues, where the position of the page was stored before setting focus and then immediately after setting focus it would scroll to that point. The customer was not satisfied with this fix as they said it did not work for modal/popup pages and for non-modal pages in IE11, there was a noticeable flicker after the focus had been set.

It was decided that the 1 second delay in focus should be removed for IE11 as it was causing a number of usability issues for all users.
This design page details in the investigation that was carried out in order to try find an alternative solution for triggering forms mode automatically in IE11 after a page has loaded and a field receives focus.

Design Options

Option 1 (chosen)

  • Remove 1 second delay in setting focus.
  • Pros of choosing this option.
    • Focus will be set on the first focusable field much faster.
    • Improved usability in IE11
    • No impact to user's scrolling actions
  • Cons of choosing this option
    • It will be difficult to trigger Forms Mode in JAWS for screen reader user when the page is loaded.
    • We have no control over JAWS.

Option 2

  • Only focus on a field when a user has not scrolled.
  • Pros of choosing this option.
    • Can keep 1 second delay and have minimal impact to JAWS users
  • Cons of choosing this option
    • No field would receive focus when a user has scrolled
    • If a user accidentally scrolls, then no field would receive focus unless we choose an arbitrary amount the user has to scroll by before deciding not to set the focus

Option 3

  • Store user's position before focus is set, and scroll back to that point before setting focus.
  • Pros of choosing this option.
    • Can keep 1 second delay and have minimal impact to JAWS users
    • Lesser impact to user's scrolling actions
  • Cons of choosing this option
    • A version of this was already given to the customer in a Testfix and they said it did not work for modals and caused a flicker on the screen in IE11

Option 4

  • Use the MicroSoft specific setActive() function to focus on the field but not scroll
  • https://developer.mozilla.org/en-US/docs/Mozilla/Gecko/Chrome/API/Browser_API/setActive
  • Pros of choosing this option.
    • No impact to user's scrolling actions
  • Cons of choosing this option
    • This function in considered non-standard and should not be used in production
    • It only worked for IE.
    • 1 second delay would still be required to activate Forms Mode.

Design

Design 1:

The one second delay in setting the focus in IE11 will be removed from the product. This has caused a number of issues since it was introduced and it's removal will boost the usability performance of the product in IE11.
It was originally added to address an issue where Forms Mode was not triggered for JAWS users when a page loaded. An alternative solution is needed to address this.
The scrolling issues MSD reported were caused by the significant amount of time it took before a field received focus, Users were able to scroll down the page before this one second threshold, and when the focusable field received focus, the page would move from the current position to the position of the focused element.

Implementation Notes

General thoughts:

Triggering Forms Mode is out of our control. We have no control overs JAWS, we can only "entice it" to behave in the expected manner.
I tried many different types of solutions in order to address this issue but none work as well as the one second delay. In my opinion, I believe the 1 second delay only works almost consistently (but not 100% of the time) because it is such a large threshold, that after 1 second all processing on the page (DOM, JS related etc) has finished or has been significantly reduced, and in that time JAWS has finished analysing the page. Then when the focus is placed, JAWS detects this and Forms Mode is triggered. At certain times the JAWS virtual cursor remains on the previously clicked element, even after a new page has loaded and the first editable field has received focus. These elements are read out correctly, but not interactive. This happened seldomly with the 1 second delay, but happens more consistently without it. I don't think we can get this working 100% of the time.
All IE11 focus related changes should be removed once support for IE11 is dropped!!

Removing 1 second delay:

I originally planned on removing it completely, and all fixes that were put into the product to address issues with the 1 second delay.
After numerous hours of testing on my own VM, on multiple streams, docker images, and sending it to Satya for early testing, it was decided that a delay was required as JAWS was acting too inconsistently.
Therefore I restored the timeouts and any changes that were put in place to address the issues with the 1 second delay (such as don't auto set focus if user has manually places the focus elsewhere)

There are now different thresholds set when focusing on certain types of fields.

Modals:
Any editable field within a modal will receive focus in 200ms
Any field that doesn't require Forms mode will receive focus instantly.
Any textarea will receive focus in 1 second as any threshold below 1 second did not work. Satya and I could not think of any use case where there is a textarea as the first element and where that page has a scroll bar.

Non-Modals:
Any editable field within a modal will receive focus in 500ms
Any field that doesn't require Forms mode will receive focus instantly.
Any textarea will receive focus in 1 second as any threshold below 1 second did not work. Satya and I could not think of any use case where there is a textarea as the first element and where that page has a scroll bar.

Hidden input field:

A hidden input field is created and focused on immediately when the first focusable field is an editable field. This helps JAWS trigger forms mode from our experience in testing.
This logic exists in both ModalDialog._setFocusHandler() and util.setFocus()
Once the onblur event is triggered on this field (when focus leaves the element), this field is destroyed. For extra safety, this field is created outside of the form incase there are use cases where the page is reloaded as soon as it loads (An example of this can be seen when creating New Evidence on a IC. There is custom JS in for the Evidence_NewEvidence.uim page that reloads the page as soon as it loads as it populates a list of evidences. The dropdown on this page also has an onchange event linked to it that will reload the page when a new value is selected)
This hidden input field is visually hidden using inline stylings and also has an aria-hidden and no labels in order to prevent it from being announced by JAWS.

To visually hide an element, I used the following stylings:
position: absolute; height: 1px; width: 1px; overflow: hidden; clip: rect(1px, 1px, 1px, 1px); white-space: nowrap;

Styling changes:

The styling changes applied as part of this fix appear to be key in getting JAWS to trigger forms mode. Before they were applied, anything threshold below 1 second did not work at all in my experience.
The styling changes for modals and tab navigation involve using opacity: 0 as opposed to visibility: hidden.
I discovered this after creating a custom webpage (I was trying to determine if it was a JAWS defect after seeing https://github.com/FreedomScientific/VFO-standards-support/issues/167? or if our HTML structure was invalid) which simulated opening a modal (I kept the same HTML structure as we have in Curam and used timeouts to simulate long page loading times). JAWS triggered forms mode consistently there even when there were no timeouts in setting focus.
I then installed a browser plugin into IE11 which disabled all styling, both inline and CSS. I noticed that Forms Mode was triggered in the Tab Test in the use cases I tried (again with no timeouts), but as soon as styling was re-added, it no longer worked. For modals, I eventually narrowed it down to the visibility:hidden styling being applied and that it was being constructed 10000px off screen. Removing these stylings addressed the issues with modals. I applied a similar fix to Tab navigation.
Unfortunately it wasn't the silver bullet as although JAWS triggered Forms Mode in a lot of cases, it didn't work as consistent as before.

Changes to popups:

I noticed that after all other changes had been applied, there were still issues with Forms Mode when opening modals from popup menus. I noticed this is where JAWS acted most inconsistently even with the 1 second delay.
I noticed that once a menu item is clicked, the menu is hidden using inline "display:none" styling, but the JAWS virtual cursor appeared to stay on the menu item. As a test, I updated the popup.js in the dojo-overrides package, to not hide the menu at all and from then on JAWS switched to Forms Mode consistently. I then decided to visually hide the menu instead of using "display:none" and this helped JAWS with switching to Forms Mode. But I discovered issues while testing, as although visually hidden, users could still potentially navigate to the hidden menu with JAWS or with the TAB button (setting the tabIndex to -1 was problematic). So I added some logic, where if the menu is not re-opened after 200ms, then the original "display:none" styling is applied to it. If it is reopened, within 200ms, then the stylings that visually hide the menu are removed.
I was reluctant to make this change as it needed to take place in the popup.js file in the dojo-overrides package. Since it is a dojo-override, it is used not only by Curam code but also by dojo code too.
I made sure that these changes only happen in IE11 in order to narrow any potential impact.

To visually hide an element, I used the following stylings:
position: absolute; height: 1px; width: 1px; overflow: hidden; clip: rect(1px, 1px, 1px, 1px); white-space: nowrap;

Note on IEG:

I investigated whether IEG was affected by the focus timeout changes in IE11. IEG has it's own javascript class that sets the focus on it's form fields, so it does not require any changes. IEG also has a timeout for setting focus and it sets the focus on element after 250ms.
The IEG focus javascript class is: /BIZINF/webclient/components/IntelligentEvidenceGathering/jscript/src/ieg/ieg-focus.js

CKEditor:

CKEditor/Rich text editors are rendered differently to the rest of our form elements. They are first rendered as textareas and then the RTE is rendered in place of that text area. The RTE is not rendered by the time the focus functions are called in order to set focus dynamically. The CKEditor code is responsible for setting the focus within the RTE. If the RTE is the first/only focusable element on the page, a flag can be passed in the UIM to set the focus on this element.
Screens where the RTE is the first focusable element may have issues with JAWS entering forms mode.

New Evidence (IC) and Transfer Evidence UIM screens:

There were additional scrolling issues with the New Evidence (IC) and Transfer Evidence modal UIM screens. This was due to these screens having custom JS applied where the page is re-loaded as soon as it first loads in order to populate a list. There is also an onchange eventlistener added to the combo box on the page, so if a new value is selected, the page will reload. This caused scrolling issues as users could select a new value and then scroll before the page had fully re-loaded and before the focus had been set after 200ms. The only way to address this was to suppress the focus delay completely for these modals. To do this, in the custom JS for the pages, I set a property in the sessionStorage called "suppressCuramModalFocusTimeout" to "true". Then in the ModalDialog.js, before setting the focus by calling curam.util.setFocusOnField(focusField, isModal, localizedModalFrameTitle), I check to see if this flag has been set. If it is set, I then suppress the timeout. After the focus has been set, this flag is then removed.
An additional change needed to be added to the custom JS where the form is submitted, This change involved wrapping the submit function in a timeout with 0ms threshold. This was done because when I tried setting the suppressCuramModalFocusTimeout flag and didn't have the timeout, I was getting a JS error saying that I couldn't "execute the code from a freed script". This usually occurs when you call a function that was created in a window or frame that no longer exists. So I believe this was happening in the custom JS because when setting a property in the sessionstorage, that property is going to be available globally and not just within the scope of the current frame. But as the page is submitted immediately after loading and therefore the frame is re-loaded, the function to set the property in the sessionstorage was not fully completed. So by setting a timeout of 0ms on the submit function, this adds that function to the end of the event queue and allows for the sessionstorage property to be set properly. This does not appear to have any impact on the functionality or UX of the page.
