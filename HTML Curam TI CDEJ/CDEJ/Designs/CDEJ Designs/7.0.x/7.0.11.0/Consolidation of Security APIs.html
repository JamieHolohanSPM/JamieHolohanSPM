Background

Contrast is a tool that the security team uses to perform scanning of the SPM code base to search for security vulnerabilities. Currently (March 2020) Contrast is not mandated by IBM but this is likely to change in the future.

Since we have started using Contrast, a number of security issues have been flagged by the tool. A team has been set-up within PD to investigate each issue, to validate, and to apply the recommended security fix as required.

From a cursory review of the code, it appears several threat mitigation mechanisms, and in some cases duplicate mitigation mechanism, are already in different locations in the code base which are being used to sanitise/mitigate un-trusted input and output data. Furthermore, some of these mitigation mechanisms, such as Active Content Filtering, have been sunset and will be removed at some stage.

The team's objective here is to:

  • Modernise our security mechanisms in order to align with current best practice
  • Consolidate these mechanisms in a central location within the product in an easy to use API.
  • Going forward, developers should familiarise themselves and use these API's to mitigate existing security vulnerabilities. In the future,  the API's should be incorporated into new secure coding practices to prevent new vulnerabilities developing.

Requirements

List of Requirements

  • A new Security component will be added to the Technical infrastructure that shall contain new security API's. These API's shall be available to all higher stacks and used explicitly for security fixes.
  • The API shall be available to both the web and server tiers and should also include the REST infrastructure.
  • The design of this new infrastructure should follow existing TI patterns that apply.
  • The Security API must provide mitigation mechanisms that align with current security industry standards and address the common security issues flagged by Contrast.
  • The API and any dependencies, such as third party libraries, will have approval from legal and located in this new central security component.
  • The API should be extensible to the customer. 
  • The API should be well documented for internal and external stakeholders.

Design

 

  • We will follow a phased approach to the implementation.
  • Phase 1. We provide a basic factory pattern. Client uses the factory to get an instance of the SecurityUtility API. This is limited to the default SecurityUtilsImpl. We are not supporting customisation of the default implementation.
  • Phase 2. The customer can provide their own API implementation by using GUICE to inject their custom SecurityUtilImpl class into the factory. This is done using an approved SPM pattern. It reuires the customer to provide the custom component and update the appropriate bindings in the GUICE module class. 
  • GUICE is new to the TI layer and we need to spike the effort in supporting customisation using GUICE.

Use Cases

This is mandatory. The various use cases for the feature must be outlined here. They can be in the following format.

 

Use Case ID 1.1
Name x-path Injection
Actor Dev
Description

XPath injection attacks occur when a website constructs an XPath query for XML data from user-supplied information. Thus, the issues that occur when using XML to store data are quite similar to those faces with SQL.
Preconditions

Contrasts scans for security weaknesses in the code and detects an x-path injection vulnerability.

Security communicates the issue to the appropriate team.

 
Flow
  • Contrasts scans for security weaknesses in the code and detects an x-path injection vulnerability.
  • Code and application logic analysis is required by the developer to see if the reported issue is valid in the reported context. False positives do occur.  XPath injection is a type of attack where malicious user input can be used to grant unauthorized access or reveal sensitive information such as XML document structure and content. This style of attack is carried out by making the user’s input be used in the construction of the query string.
  • Security Documentation will guide the developer to the correct API -- in this case the compileXPathExpressionPreventXPathInjection
  • compileXPathExpressionPreventXPathInjection -- The developer will use this method, so that queries are precompiled and thus pass user input as parameters instead of expressions. The resulting x-path expression is used to search the xml document.
Expected

Returns the XPath expression.

If invalid, an XPath expression exception & Application exception thrown.

Application Impact

  • Many of the new api's that we will add to the security-inf component will likely do some form of data sanitisation:  validation, encoding, escaping, removing etc. This has the potential to impact application performance and functionality.  As we develop out the API and start to use it in the the code base, we need to routinely consider any impact. Robust testing will be required both of the API and its use within the code base. Testing should encompass both J-unit , CPT by QA and PVT by performance team.
  • A number of security features have been added to SPM and overtime they have become scattered throughout the code base making them difficult to find and maintain. With this new security component, the plan is to move and consolidate the features in one location. This refactoring may have an impact on functionality. Careful analysis will required as each feature is moved.
