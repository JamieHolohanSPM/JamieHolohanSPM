Background

Connections and various other IBM products have been divested to HCL. As part of this process, two libraries used in SPM (acf and htmlparser) were also included in that bundling. This means that for SPM to continue using these libraries in the product, some fee would have to be paid to HCL.

Therefore, a decision has been made to remove these libraries from the product and look for alternatives to perform the same functionality as the libraries provided.

The most recent communication on this was via email on 29-01-2020 where a bundling dependency on ACF triggered an email conversation with Duinseach Carey/Ireland/IBM. The below text is the result of that communication:

I've confirmed there is no new owning team for ACF at IBM. Looking at the list of dependencies in CH, most are HCL now (Connections, Docs, Sametime, Portal). That leaves Curam and Rational Requirements Composer, which looks to have gone EOL in September of 2017, so essentially Curam is the only active consumer of the code.

HCL will be independently maintaining the version they own and use (they have the source and rights to continue with it) so that leaves your team with two choices - either maintain your copy or move off to an alternative. My understanding is that other teams such as Watson Health are planning to move to OWASP so that may be something you want to consider? I can accept this last dependency request based on your agreement that continued use of the code will be at your own risk: it is unsupported, there will be no further development and no bug fixes. Beyond this request, unless your team wishes to take ownership of the record, I will be following up with the CH team re getting it moved to EOL.

It should also ne noted that that ACF has been marked as EOL within Clearing House and hence no future bundling dependencies can be raised against it:

  • https://w3-connections.ibm.com/forums/html/topic?id=aad686b4-04c5-4e9c-a413-c667f7be2737

Requirements

  • Requirement 1: Remove the acf and html parser JARs from the SPM deliverable.
  • Requirement 2: Replace the functionality that these JARs provided and usage within the product with an alternative technology so as that there is no impact to SPM and the result of any processing performed by these libraries is the same.
  • Requirement 3: This removal exercise must be performed across all of the live versions of the SPM product.

Discussion on Usage and Possible Replacement

I have reviewed our use of this within `curam.widget.render.utils.SecurityUtils.java`, which is where I worked on that defect previously.

I think what we are doing with the call to ACF is to set up a blacklist of specifically prohibited characters, and apply it to a document that has already had a whitelist applied. So the flow within that filterACF() method is as follows:

  1. Set up the processor factory and instance for manipulating the document.
  2. Check if whitelist is enabled (system configuration)
  3.    a. If it is, read whitelist from properties file and apply the whitelist using jsoup.
  4.    b. If not break out of that check.
  5. Apply the blacklist using ACF.

I think we should be fine to take the same process and replicate in jsoup, so it becomes

1. Check if whitelist is enabled (system configuration)
   a.  If it is, read whitelist from properties file and apply the whitelist using jsoup.
   b. If not break out of that check - and apply blacklist, applying it to the document once it has already been manipulated by the whitelist is redundant, _and_ raises an issue where a customer enables a tag for whatever reason in their propertiesfile, but we have magic code in ACF that removes it, with no visibility.

Jsoup doesn’t really deal with the idea of a `blacklist`, only whitelists. I would question the use case of applying a blacklist at all actually, I remember Fergus wanted all uses of it removed over time. We would be best off filtering all content through a whitelist and a only through a whitelist, the blacklist is redundant with modern injection/obfuscation techniques.

But what we could do is create a “Whitelist” that allows all tags, and then remove the ones that are currently on the blacklist in that ACF code, and if whitelist is not enabled by that system config, apply that more lenient whitelist. It would have the same effect.

 

 /**
  * This method takes untrusted html, and filters through a white-list of
  * permitted tags and attributes (designed to match rich text editor
  * white-list) before returning the safe HTML.
  *
  * @param richText The html string to be filtered.
  *
  * @return the sanitized html string.
  */

This is the sanitize method, which used Jsoup to clean the passed in html based on a very restrictive whitelist. Basically, we start off with Whitelist.none(), which is an empty whitelist, no tags will be allowed at all. We then read the whiteListMap, and apply the tags and attributes from that to our whitelist, and then apply the whitelist to the html.

We only call this method if the `kWhiteListFilterDisabledConfig` system property is not set.

Regardless of whether or not that property is set, we currently use ACF to apply a blacklist. You can see the flow of this in my next message.

The ideal solution here would be to enforce whitelist usage by removing that system property, and remove any use of and reference of a blacklist, as they are a antiquated,  allowing all tags by default except those defined by the blacklist.

An attacker could use _any_ of the methods here `https://www.owasp.org/index.php/XSS_Filter_Evasion_Cheat_Sheet` to still inject malicious code past this blacklist.

The whitelist approach, used within the sanitize method, lets nothing through, except that outlined in the properties file,

 

Check for system configuration before applying whitelist. Always apply blacklist.

Replacement Options

 

Option 1 - Java HTML Sanitizer

A replacement for the ACF tool that HCL are looking for licensing fees for may be the following. Turns out we probably should be using this one (https://github.com/owasp/java-html-sanitizer ) anyway as the ACF one had fallen behind. It appears to do all that we require, so there should be no issue migrating to this tool, but as always these things take time and because of turn over in staff there is unlikely to be anyone overly familiar with the these tools and current implementations in the product.

Pros:

  • Seems to be current.
  • Seems to do what we want.

Cons:

  • New OSS to introduce and maintain.
  • Fit gap analysis required to determine if it satsifies our current requirements.

Option 2 - JSoup

We have used this library before in other places for this purpose.

Pros:

  • Already in use so we know how it functions.
  • No requirement to add in new OSS into the product.

Cons:

  • Need to determine if it will satisfy our current requirements.

  • Will the user be able to take corrective action in the system without the need to resort to application logs/SQL searching/data manipulation outside of the system.
  • Will the data be left in a consistent state within the system after the exception is thrown?
  • Will there be messages given to the user? What will these messages be?
  • Will there something written to the logs? What will this be?
  • Does the feature require some of the exception data to be captured and an admin function provided to allow the situation to be re-mediated?

Design Logging

Document any logging requirements for the new feature being added. Does the new feature require some logging output to aid debugging in a black box environment? Think about what the logging could be to avoid revealing PI information but giving enough information to allow effective debugging to occur in the event of an unexpected issue? JMX considerations should also be added here.
