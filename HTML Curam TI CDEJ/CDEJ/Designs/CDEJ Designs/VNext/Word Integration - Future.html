Word Integration - Future

Design Page for - Word Integration - Future

Background

What is it?

  • Allows a user to edit a Microsoft Word document on their local computer and then save it to the IBM Cúram database.
  • Also an additional feature within the Appeals component which allows a clause to picked and automatically inserted into a Word document.
  • A document can be created automatically from a template where the template details can be set before the document is presented to the user for editing.
  • The FILE_EDIT widget uses either a Java applet to manage the interaction between the user's browser and Word (for the browsers which support Java e.g. Internet Explorer) or the Native Messaging API for Chrome.

  • The customers who use this feature like it. Some of these customers are listed below:

    • Alberta

    • SAMS

    • CPIN

    • DSCIS

    • Hamburg

Recent History

  • Pre version 6.1 of the product, there were many ongoing issues with this feature.
    • Performance issues.
    • Instabilities when using COM bridge (findWindows).
    • Outdated, for one-windowed/popup based application.
    • Multiple APARs from customers which proved difficult to reproduce, diagnose and fix due the number of moving parts within the overall feature (browser version, JRE level, Word version etc.).
  • Chrome then announced they were dropping NPAPI (Java, applet) support.
    • http://blog.chromium.org/2014/11/the-final-countdown-for-npapi.html
  • After some discussion, this led to the building of a “Chrome specific” solution for the feature (included a stand alone Java process and a Chrome extension).
  • Also led to re-writing of the infrastructure to support the Java widget for Internet Explorer to ensure it was more stable and that the user experience was better.
  • Overall, 4-5 months of effort to achieve this.
  • Now have two separate implementations, one for Chrome and one for Internet Explorer.
  • The Widget has proved relatively stable since it was re-written/re-factored for the 6.1 release.
  • There are still some customer cases coming in but not of the same volume as pre the re-write.
  • There are some still edge case issues such as the Word application is left open when the session is terminated unexpectedly.

File Edit Widget - What is Next

  • There have been recent announcements from Microsoft that they are going to follow suit and drop the Java plugin support in their browsers in the future.
  • Oracle have also announced they are going to drop the browser plug-in support starting with the Java 9 release.
  • These changes have already begun with the Microsoft Edge browser (Windows 10) dropping support for running Java in the browser (applet support).
    • http://answers.microsoft.com/en-us/windows/forum/apps_windows_10-msedge/ms-edge-plugins/eb4e3bed-1a0b-4d95-b437-8fdbb4482691
  • This change in third party policy by Microsoft means that the current file edit solution will not work on Microsoft Edge (we have ran a quick test and found this to be the case).
  • In addition to that, Microsoft have not yet proposed any alternative to the old functionality and at this stage, Edge is a complete black box with no allowed interaction.
  • Currently therefore, we could not, even if we wanted, build a bespoke alternative solution for Edge in a similar fashion to Chrome.
  • What are the alternatives?
    • Drop support for this feature altogether from the product.
    • Continue support for the feature in Chrome/IE and just state it will not work in Microsoft Edge.
    • Provide an alternative solution to solve the customer’s business use case.
    • Download/Upload Word Document using existing widgets.
    • Investigate open source alternatives?
  • Further investment in this feature v the benefits. Discuss?

Support Matrix for the File Edit Widget

  • There are a number of variables to be considered in a support matrix for this feature:
    • Cúram version.
    • Browser Type and Version.
    • JRE Version.
    • Two current solutions:
      • Java applet and the COM bridge combination for the browsers which support applets (IE)
      • Native Messaging Bridge (Chrome Extension + Standalone Java Client) + Modified COM bridge for Chrome.
  • Some significant events (some past and future) also have to be fed into the support matrix:
    • Chrome browser discontinue support for running Java in the browser (Sep, 2015).
    • Announcements from Microsoft that they are going to follow suit and drop the Java plugin support in future. Latest browser from Microsoft (Edge) already does this.
    • JRE 9.0 – Oracle announce that this version will drop browser plug in support (Sep, 2016).

Non Functional Requirements

This is mandatory. We can fill in N/A if the specified non functional requirement is not applicable to the feature being delivered, otherwise some notes must be entered here covering the particular area.

 

NFR Note                                     
Accessibility The ability of the system to be used by people with disabilities. When new technologies are being used for the UI very careful consideration should be given to whether screen readers can be used with the technologies. This applies specifically to usage of JavaScript, Flex and SVG. The use of colors to provide functionality should be considered (and avoided), along with resolutions that will be used by vision impaired users.
Availability & Resilience The ability of the system to be fully or partly operational as and when required and to effectively handle failures that could affect system availability. This should be considered when working outside of the normal infrastructure and deployment. For example, the XML Server is not deployed in an application server and has special handling for fail-over and load balancing. Careful consideration should be taken before adding new components, e.g. EJB Beans, MDBs, to the infrastructure.
Development Resource The ability of the system to be designed, built, deployed and operated within known constraints of people, budget, time, etc. In general there is not much information required here, however where special skill sets or people with siloed knowledge are required to work on the change/enhancement, this should be detailed here.
Evolution The ability of the system to be flexible in the face of the inevitable change that all systems experience after deployment, balanced against the costs of providing such flexibility. Consideration should be given to future proofing the change, including considering how to upgrade to any changes or enhancements provided in the future. Specifically designs should accommodate elements such as localisation (text, dates, etc.), customization, configurability, and componentization. Providing these features in future releases, when not considered up front, has caused impacts for customers in the past and has prevented us from providing the features in some cases.
Internationalisation The ability of the system to be independent from any particular language, country or cultural group. Localisation and internationalisation are a large concern for Cúram. Support should always be provided to view information in multiple locales, based on the user and server locale, along with support for working in multiple time zones.
Location The ability of the system to overcome problems brought about by the absolute location of its elements and the distances between them. For the most part this is not a major concern, however we should never design based on the assumption that the customer will use a default configuration
Performance and Scalability The ability of the system to predictably execute within its mandated performance profile and to handle increased processing volumes. Performance and scalability are a concern and should always be considered during design. Where necessary, the Performance and Benchmarking teaming should be engaged early to help with determining impact of potential changes.
Regulation The ability of the system to conform to locale and international laws, quasi-legal regulations, company policies and other rules and standards. Regulation includes the usages of third party tools and the licensing issues associated with these. New jars should never be introduced without considering the legal implications and licensing of the jars. This includes usage of GPL licensed code. Also, usage of code freely provided on the web and not properly licensed should be avoided.
Security The ability of the system to reliably control, monitor and audit who can perform what actions on what resources and to detect and recover from failures in security mechanisms. This is a concern where new features are added that are not contained within the normal Cúram security.
Usability The ease with which people who interact with the system can work effectively. How the change will affect the user and how they use the application/feature should always be considered. Where necessary the business analysts and our user interface experts should be engaged early to ensure we meet usability requirements. e.g. printing.

Resilience


This is mandatory. Some thought should be given to how resilient the new feature is.

  •     What happens if something expected/unexpected goes wrong?
  •     What happens to the system?
  •     Will the user be able to take corrective action in the system without the need to resort to application logs/SQL searching/data manipulation outside of the system.
  •     Will the data be left in a consistent state within the system after the exception is thrown?
  •     Will there be messages given to the user? What will these messages be?
  •     Will there something written to the logs? What will this be?
  •     Does the feature require some of the exception data to be captured and an admin function provided to allow the situation to be re-mediated?
