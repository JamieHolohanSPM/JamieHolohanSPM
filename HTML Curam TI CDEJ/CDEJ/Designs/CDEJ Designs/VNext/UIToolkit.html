Provide a toolkit that enables developers to build quality user interface. See here for more thorough explanation of the project: http://dubxpcvm065.mul.ie.ibm.com/EJB/TechInf/ClientTeam/ArchitectureAndDesign/UiToolingProject.html

Design Options

The UI Toolkit should be a black box toolkit that comes as a separate deliverable.

  • Option 1

    • Create the UI Toolkit as part of the JDE. The JDE ant scripts provide the required functionality.

  • Option 2 (preferred)

    • Create a separate UI Toolkit deliverable (similar to ADBuildTools) that will be pulled into developer's streams. The toolkit should have ant scripts that developers that wrap JDE ant scripts that provide the required functionality.

Design

The toolkit is basically some build scripts that has dependencies on third party software i.e intern, dojo e.t.c and the cdej source. The build scripts will wrap JDE scripts where applicable.

It is envisaged that this toolkit will be brought down as a new deliverable from the network into a directory called UIToolkit in developers development environment.

We currently do not envisage that the UI Toolkit would be shipped to customers so having a separate deliverable makes sense. A big advantage of a separate UI Toolkit deliverable is that we could release new versions of the Toolkit between JDE releases if we wanted to (e.g. new version of Intern/Selenium e.t.c)

UI Toolkit Delivery mechanism

    Put the text files that control version of third party software and build scripts on the CuramRelTools VOB so that a separate can be delivered similar to ADBuildTools.

    Version the UIToollkit deliverable (zipped deliverable) on DevReleases directory on network similar to other releases.

(2) Command line to CEF's Base Folder

(3) Command line to webclient folder: cd webclient

(4) First time configuring intern for a component project in the client:

run 'appbuild.bat new-intern-suite -Dcomponent=<<component-name>>

NOTE: 'componentPath' is optional parameter outside of Curam Context..

eg.: appbuild.bat new-intern-suite -Dcomponent=core
 

This will build the intern configuration file and output them to the %UITOOLKIT_HOME%\tools\intern\generated\<<component>> directory.


(5) If this is the first time you are configuring intern then copy the generated 'intern-config' directory to webclient\components\<<component>>

For example if you want to unit test JavaScript for the core component, then copy to webclient\components\core


(6) Deguging tests in eclipse.

    Refresh the core component in eclipse

    Go to the 'External Tools' menu in eclipse.

    From the dropdown in the 'External Tools' menu, select 'External Tools Configurations' and then underneath 'Program' menu, select 'debug-intern-tests-core'.

 This will open up an instance of a chrome and run the tests in debug mode. You should open the developers tools (F12) and ecnsure no errors on console.

 NOTE:

 (i) You can set a breakpoint at any place within your tests or javascript on developers tools console.

 (ii) If you want to debug in another browser , just open an instance of that browser and copy the url in, the tests wil then be run in that browser.

(7) Running tests/coverage report

The tests (& covregae report) can be run on the command line or in eclipse.

(A) To run tests and report code coverage in eclipse (CURRENTLY NOT SUPPORTED)

Go to the 'External Tools' menu in eclipse.

From the dropdown in the 'External Tools' menu, select 'External Tools Configurations' and then underneath 'Program' menu, select 'run-intern-tests-core'.

Documentation/How To Use Feature

Prerequisistes for JavaScript Unit testing

  1. Enforced JavaScript structure per component.  The following structure must be followed within each component.

            jscript

                -> src (recommended source code location)

                            -> curam (recommended root namespace)

                 -> tests (recommended)

     

  1. New AMD Format. The Javascript modules must use the new AMD format: http://dojotoolkit.org/reference-guide/1.8/loader/amd.html#loader-amd.   The modules cannot be in legacy format http://dojotoolkit.org/reference-guide/1.8/loader/legacy.html#loader-legacy.
  2. Dojo V2 Dom Modules. Most deprecated dojo modules can be used with the current version of intern that is being used. However the deprecated dom modules cannot be used. The new dom modules must be used instead. Please refer to the quick reference here for old syntax (V1)versus (V2): http://livedocs.dojotoolkit.org/releasenotes/migration-2.0#id4
  3.   Java Version. Java version must be at least JDK 7.

How to use feature

The very first time you want to write/run JavaScript unit tests against a webclient application component, then some configuration will be required but this will only take a minute or two. Once a component has been configured to run JavaScript unit tests then the tests (and coverage reports) can be run as often as required.

 

(1) JavaScript Test Configuration.

In order to configure a component to be able to run JavaScript tests, open a command prompt and navigate to the webclient directory. Then run

run 'appbuild.bat new-intern-suite -Dcomponent=<<component-name>>. For example if configuring tests against core component then you would run 'appbuild.bat new-intern-suite -Dcomponent=core'.

 

After you run this command, two directories will be generated:

(i)  intern-config. It will get generated in %UITOOLKIT_HOME%\tools\intern\generated\<<component>> directory. This contains the configuration of your JavaScript source and your JavaScript test files. You need to copy this to your application component that you are configuring. For example if configuring core component then you need to copy this directory (intern-config) to the root of the %CLIENT_DIR%\components\core directory.

(ii) jscript. It will get generated in %UITOOLKIT_HOME%\tools\intern\generated\<<component>> directory. This contains a skeleton JavaScript tests suite file that will enable you to get up and running quickly with the tests. e.g jscript/tests/coreTestSuite.js. You need to copy this to your application component that you are configuring. For example if configuring core component then you need to copy this directory (jscript) to the root of the %CLIENT_DIR%\components\core directory.

 

Assuming that you have followed the recommended directory structure for JavaScript source code and test code then nothing further is required. However if your source are tests are in a different location then you will need to update the 'packages' configuration as appropriate within the intern-config.js file.
