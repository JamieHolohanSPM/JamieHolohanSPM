Background

The Curam application is pretty dynamic in so far as that it has many UI widgets that has JavaScript logic that changes the behaviour of the UI in response to the user performing some action.

 

These are essentially 2 ways of including a JavaScript file (module in AMD terminology) onto a web page in order to give the UI widgets on that page the dynamic behavior.

(1) Renderers

There are two ways in which JavaScript can be included in on a webpage via a Java Renderer

(i) By using the includeScripts() method of The RendererContext Object which can be used to set variables (also global variables), include a plain JavaScript file or include a Dojo Module (Module is Dojo terminology for a file).

(ii) By setting the "dojo-data-type" attribute on an element. This is ONLY specifically done for Dojo JavaScript widget also known as a Dijit Widget. When the Dojo parser in invoked at runtime it substitutes the  element that sets the attribute for the HTML template of the specified Dijit widget. For example when the following Div sets the attribute like so menuItemDiv.setAttribute("data-dojo-type", "dijit/Menu"); that div is replaced by the HTML for the Menu widget at runtime.

 

(2) UIM

Currently JavaScript can be referenced in a UIM file using the SCRIPT_FILE attribute. When UIM is converted to JSPX files during the build process 2 things happen to enable the JavaScript to be invoked on a web page at runtime:

(i) The JavaScript file from the component gets copied into the root of the CDEJ/jscript (name space is NOT respected) within the ClientModule.war file. An important caveat here is the JavaScript file must NOT be in a WebContent directory of a component in order to be used directly in UIM.

(ii) The JavaScript file gets pulled into the web page by including it on the generated JSPX file like so - where <<fileName>> is the name of JavaScript file referred to in JSCRIPT_FILE in UIM:

 <script src="../CDEJ/jscript/<<fileName>>.js" type="text/javascript">// dummy script content</script>

 

The key points here are:

(1) When including JavaScript via a renderer the JavaScript can be included by referencing the file from the root  (CDEJ/jscript) OR by referencing the properly name spaced file relative to the root. Whereas for UIM, ONLY the JavaScript file at the root can be referenced.

(2) In order to get a JavaScript file generated into the root, the JavaScript file in source can be placed anywhere EXCEPT in a WebContent directory within a component. In order to get a JavaScript file generated into the properly name spaced location, then the JavaScript file in source MUST be placed in a WebContent directory within a component.

 

Requirements

The actual requirement here is to support the inclusion of AMD Dojo modules in UIM and in renderers. The word AMD is important here. It means the code is in Asynchronous Module Definition format. More information on this the Dojo AMD format can be viewed here: https://dojotoolkit.org/documentation/tutorials/1.10/modules/

 

In the context of this requirement, an important question is, why do we need to think about this now because the AMD format has been around for the last 5 years or so?

The ultimate driver for this requirement is that AMD will be the only format supported for version 2 of the Dojo Toolkit. The Legacy format will NOT be supported for Dojo version 2.0. This means there is a time limit on how long long we have to convert legacy Dojo modules into AMD modules (assuming we still want to support Dojo in the future). A secondary driver is that if application developers want to unit test Dojo modules (we don't care about plain JavaScript files with no Dojo) using the UI Toolkit then they need to convert the legacy modules into AMD modules. Therefore we anticipate that in the near to medium term developers will convert legacy modules to AMD modules.

 

An other important point bearing in mind with respect to this requirement is that because there is a mix of AMD modules and legacy Dojo modules in the application, Dojo is loaded into the application using the legacy loader instead of the AMD loader. The AMD loader loads JavaScript asynchronously and thereby improves performance. See here for more information: https://dojotoolkit.org/reference-guide/1.10/loader/amd.html.

All of the Dojo modules need to be converted to AMD modules in the application, before the AMD loader can be turned on.

 

So what needs to be done in order to support AMD Dojo modules in UIM and in Java Renderers (assuming that we are sticking with Dojo as one our supported JavaScript frameworks in the future)?

I think the key to answering is to consider the following:

  • Do we want to turn on the AMD loader now (after all legacy modules converted) or in the future?
  • Do we want to preserve name spaces for developers? Currently in UIM when including Dojo files on a page, the full namespace for the file is not properly respected.

 

List of Requirements

  • Requirement 1: Validate the AMD Modules to ensure that modules written conform to what we expect
  • Requirement 2: Update the UIM generator to require modules asynchronously
  • Requirement 3: Validate the invocation of Dojo in Java Renderers to ensure it the Dojo code is invoked the proper AMD way so that the AMD loader can be turned on. This requirement is dependent on whether we would like to enable AMD loader or not!!
  • Requirement 4: Update the TI Java Code (not Java Renderers) so that all Dojo modules invoked in AMD way (not legacy). This requirement is dependent on whether we would like to enable AMD loader or not!!

Design Options

OPTION 1: (Story 197656) Validate the declared name of Dojo AMD Modules and update generator to require Module

  • This means that we load up all the Dojo modules (legacy format and AMD format) using the legacy loader but enable AMD modules to be referenced in UIM and Java Renderers.

             

The following work would be needed:

       - Document explicitly what a developer must do when converting a non AMD module to an AMD module.

       - Write an AMD module validator that does certain validations on and existing AMD module that exist in the recommended new structure i.e jscript\src

       - Update the UIM generator to require a Dojo module in correctly

        i.e     <script>require(["<<fileName>>"], function(){ });</script> instead of the old way  <script src="../CDEJ/jscript/<<fileName>>.js" type="text/javascript">// dummy script content</script>

           An important consideration when doing the above is whether at this point we want to preserve the namespace at this point as we are already copying the JavaScript the full name space already into jscript directory at root of WebContent

  • The Pros of choosing this option are:

             - Not much work needed

             - We are doing the minimum necessary in preparing to migrate to Dojo 2.0. This could be a wise approach because we do not want to invest too much on Dojo up front considering we are not sure that this will be the JavaScript framework toolkit we will be using in the long term future.

  • Cons of choosing this option

             If we do not consider enabling the AMD loader to be turned on now then perhaps we risk not giving the customer advance notice about what needs to be done before we can turn it on.

OPTION 2: (Story 197656) Enable the AMD loader to be turned on as soon as possible

  • This means that we do the work now up front to enable the AMD loader to be switched on as soon as possible (when all modules converted to AMD format).

 

             The basic design is that there would be a build time property available that enables or disables the Dojo AMD Loader to be turned on. The loader would be turned OFF by default. In this scenario the client would build as normal.

 If however the property was ON, then the design option here proposes that we need to implement 2 things:

   -  A validator that validates Java renderers and Dojo modules to ensure that they invoke Dojo correctly in the case of the former and that the modules are correctly formatted for the latter.

- The UIM generator needs to be updated so that Dojo script are required properly. This is same a described above.

  • The only con really is the amount of work involved. It is quite a lot of work especially if we decide to go with another JavaScript framework in a few years time
  • The pros of this approach is that all the work for AMD loading is now done up front. There will be  nothing more to do with repect to AMD loading design.
