Top level window services

Our tabbed application has been built up on the single page application by changing the context these single pages load.

For a single page application every loaded page is a separate entity and must contain certain JavaScript code and certain data in order to work properly. Therefore, we have lots of code loaded every time a page is navigated to.

Loading a page is also associated with number of network requests to get back to HTML/CSS/JavaScript and images used on it.

 

Since the tabbed application might contain a number of the pages loaded at the same time (separate tabs/tab parts/tab furniture) , each of them in effect loads into a separate browser window (iframe). These frames co-exists, each adding to the footprint of the application.

Even though there are means to limit the potential number of simultaneously opened iframes (by controlling the number of open tabs), they are partial, since the further iframes, not related to tabs but embedded on a page as the expandable list row contents aren't limited.

Therefore, if we cannot, at least for now, avoid the iframes for the single pages, we might try to minimize the content loaded in each of these frames.

The network load is attempted to minimize by the browser itself by caching the required files; from our side we pull together the related JavaScript code and load the always needed Dojo/JavaScript as a single layer - therefore just issuing one request instead of multiple ones, and due to the browser caching getting the copy of that file from cache.

However, this is where our savings end. Each time the JS file is requested, it needs to be parsed and a separate copy of dojo registry filled in.

Once a page unloads, the already parsed and initialized code is lost and needs to be loaded, re-parsed and re-initialized for the next page.

This is clearly a waste, causing lower performance and the already mentioned bigger footprint.

One other thing to be mentioned is the JavaScript directly output on a page as part of inline scripting elements. There is no caching at all happening for the inline scripts, they are loaded as part of each page, yet their copies are used across the application; while these scripts might be considered small, they have their effect on performance (need to be parsed), network load and footprint and the side effect of reducing the readability of the generated HTML.

 

It might be inevitable in some cases, however looking at the contents of a single page one can notice the data and methods which do really belong to the application rather that a single page.

This data is the candidate to be converted to the shared code which would be

a) loaded once

b) loaded on the first demand.

The tabbed application has a clear advantage over a single page application in that it has the separate, context page which stays in place for the duration of the application usage. The page is called AppController.jspx (application.jspx), and currently contains certain code related to the tab loading and management (UIController).

It does also have a separate layer file which contains a different subset of JavaScript modules, just meaningful in this background window context (but hugely overlapping with the main client page scripts).

This entry or top level page (or pages taking into account application.jspx as well) is to an extent under-utilized taking into account its 'persistence'.

Top Services API

Therefore, the task emerges to review the code currently residing on a single page and come out with the approach of moving that code to the top level window, making it available to every page via a well defined and extensible mechanism. At the moment lets call that mechanism TopServices API (TWS).

The idea in the technical sense is to have the top level window initialization changed in such a way that it creates some sort of own JavaScript registry, where the shared code is contained under a distinct namespaces - services.

These services can be either preloaded in advance if they are immediately and inevitably needed, or upon the first request from a dependent page via the already mentioned API. Also, depending on the application mode and user preferences, the additional, optional services (like performance measuring or logging) might be loaded by changing the browser settings and persisting these settings in the browser local storage (which should cause the application to reload, adding the requested services to the top registry).

Furthermore, every (dependent) page loaded in the top window context notifies its loading fact to the top level window, and in response receives the access to the top level window reference and the services currently available in it. This allows the page to request the additional services or use the existing ones (or obey the application mode, say requesting the aspects requiring for performance measurement).

Additionally, there are lots of requests for the top level window from a single page JavaScript code, which usually traverse the DOM to find it repeatedly, and which would not longer be needed if the top window reference is available to a page from the very beginning.

The areas which might be converted to the top window services

  1. performance measurement
    Performance measurement infrastructure is the part of the wider inspection infrastructure, an optional addition which should be possible to add to the application runtime by dropping into the agreed location and turning on from the browser (and turning off once not needed).
  2. Logging/debugging
    Currently the debugging logging statements clutter the code and sometimes slow it down even when the logging is not required/switched off. The idea is to convert the logger to another inspector in the inspection infrastructure so that it becomes
    a) dynamically turned on/off;
    b) fully additional to the main functionality, no longer integrated, a layer loaded on demand
    c) potentially adaptive so that only the invocations of interest are logged.
  3. User monitoring service
    In order to adaptively plan the development effort and further enhancement areas, user statistics are useful; these statistics might be collected at various levels, however, mostly when the exchange between the server and browser happens. There is gray area when it comes to the user interface usage, and this information might be useful - for both the application development and the customer staff training (by identifying the areas problematic to navigate or understand or use). The inspection infrastructure and top window service which in fact would be a dedicated and operations bound logger with the specific reporting would be another useful addition to the system.
  4. Runtime / hot patch testing
    There is the JavaScipt testing framework already in place, this is not seen as a substitution, but rather a supplement, optionally available as a layer on top of the main functionality and depending on the structure useful to test the particular method input and output. Depending on the purpose it might also be used to selectively hot replace certain methods/invocations to preliminary test the fixes/changes without actually changing the main functionality.
  5. Testing enhancements (proof of concept)
    The extension of the previous idea, might again be a separate inspector.
  6. In-place domain validation
    This functionality is currently commented out, however it an attempt to reduce the network traffic it might be re-introduced, made optional and allow for early reporting of the input mistakes; coupled with HTML 5 validation features it might prove an useful addition in the new version.
  7. Popup management
    Each page, with or without popups associated loads up the known popup configurations and the associated handling code including the 'popup registry' for the value exchange. This could be reworked first of all, and second the handling should be moved to the top level window services.
  8. Modal dialog management
    This would be one of the most important application in top level services. The modal dialogs are managed by the top level window anyway, however the balance of the functionality is not even, scattered and also has some functional limitations (the well known double submit problem or the fact that the old modal dialog needs to be left in memory until another one loads). The management of the flow upon the modal closing is tricky and not evident. Converting the modal dialog functionality into the top level window service would allow for code concentration in one area, decouple the data and methods which are not relevant to the modal window contents or the background page contents.
  9. Error messages and informationals.
    The error messages and informationals are handled per page, either upon form submission or upon page load. There are situations, however, when these messages are cleared, or, to avoid them being cleared they are pushed to the local storage and then attempted to re-display; because of the page flow specifics this mechanism is tricky and prone to bugs. Having the top level service which would help managing these messages would help reducing the mess and allow for additional flexibility.
  10. Date/other formatting
    Each page, with or without date fields contain the data which allows the proper date formatting and handling. This is, however, clearly excessive at the page level and should be converted into the top window service.
  11. Localized messages
    There are locale-dependent text messages which are loaded and stored in every page context, some of them never used. These text blocks might be loaded on demand and once, and stored in the top level window, being served from this central location. The user locale does not usually change per app usage, and even if it changes, this change requires the application to be reloaded, hence localized messages as a service would not cause ant locale mismatch across the pages.
  12. Screen context. Each page has a separate screen context object, which allows for loading that context related code. This data is coming from the request, then converted to the JavaScript object and used when handling the links. The JavaScript object might be moved from a page to the top level window and converted to be a service, thus allowing for the centralized management and also additional logic which is difficult to implement in a separate page context.
  13. Tab configuration and refresh data.
    The advantage of having the controller page is that the code and functionality might be reused. One of the types of the information which is not reused is the tab configuration and tab refresh related data. There are tabs in the application which logically share the same configuration and might be loaded much faster if that fact was utilized, and the configurations were appropriately cached and reused just focusing on the possible dynamics specific to a tab instance (like enabled/disabled/hidden navigation tabs) . The overhead associated with that is that the tab configuration is read from the database and parsed every time that type of tab is requested, while if it was cached and managed by the top level window, this information would be ready to use at the load time.
    Same applies to the tab refresh configurations.
  14. Accessibility related data
    Accessibility related data is part of HTML. While it is not automatically considered to be optional, it is the additional data added to the DOM and as such might be subject to optional, layered loading and serving from one centralized location instead of being initially added to the HTML. This is just an idea at this stage and might be dropped if turns out to be not feasible, however it is seen falling under the same approach.
  15. Unique ID maintenance
    There is currently a work aimed at introducing the unique identifiers for the DOM objects on a page, mainly for the testing purposes. This potentially could be done as a top level window service, making it optional and tailored for different purposes/different levels of identification.
  16. Data related to the collapsible clusters/paginated lists/expandable lists and some other page-level registries which are loaded by design but optional by usage.
    This is the preliminary area to review and investigate. Some of the code might already be optional and some of it isn't necessarily convertible to the top level service (or such a conversion would be too expensive for the effect achieved).

The requirements to the TWS API

  • Initial minimalism to allow for quickest possible initialization.
    • The initial services code should load in the synchronous mode as soon as dojo framework becomes available. To achieve maximum efficiency, the loading mechanism should combine the inline scripting (output static elements from Java), the basic functionality as one module and the optional functionality as a separate module, loaded asynchronously; any parts which might be bound to some later activity should be required by that activity as an additional services.
    • User configuration support should be loaded as an optional service, on demand, as a prerequisite for the configurator (modal window or background screen).
    • Overloading might be also considered if proves more efficient: the synchronous first load engages shorter, minimalistic version of the API implementation overridden by the full versions of the same methods later where more conditions and variations become relevant; this way the API is not only amended to asynchronously, but built and finalized the same way.
    • It should be possible to unload or cancel the unnecessary services, not only explicit but also implicit, based on the last access so that the rarely used services are not kept in memory.
  • Maximum flexibility and extensibility to allow for easy addition of the new services, easy and clear their configuration; the potential to be used in a single page application (to be future proof) so that the same concept could be ported easily and would not prevent loading the pages outside the tabbed context for the testing purposes.
  • Maximum dynamics by allowing for choosing the right number and content of the optional services and the mode most suitable to both the user and the system supervisor needs: the optional services should be switched on and on dynamically and configured both by the global settings (per application) and locally, by the settings stored in the browser - with local settings overriding the global ones.
  • It is crucial to distribute the functionality and foresee the right interaction, exchange and dependency determination points and stages.
    • Say, some service related code might be bound to the single visible window load event thus plugging into the existing 'onload' infrastructure to ensure the right loading time when these services become relevant.
    • Maximum controlled inter-dependency, where both the client window might request the additional services, specific to the page/pages/application and the top window, depending on the current configuration might change the prerequisite modules of the pages being load, thus ensuring the true load on demand.

Services 'contract'

1. Top page loads

  1.    TWS statics are output as the inline JavaScript (Java output), topvars (application-wide variables) are initialized.
  2.     Services (manager) and the basic loading infrastructure is synchronously loaded.
  3.     Mandatory services are loaded (synchronous and asynchronous load combinations and service order should be considered),
  4.     Optional TWS module load is initiated.
  5.     Pre-configured optional services load is initiated.

2. A client page loads

  1.    the client page notifies top level page initiating exchange channel.
  2.    top level page injects the application wide variable references into the client page.
  3.     top level window notifies the client page about the prerequisite modules required for the loaded services
  4.     client page loads the required prerequisite modules

3. User interaction/other events on the page happen

  1.     client page requests a service (data) from the top level window
  2.     top level window looks up the services registry and:
    • either returns back the data (after processing whatever needed) to the client page
    • or loads the requested service if its isn't yet loaded and then returns the data to the client page
  3. if necessary for the requested service, top window also initiates the additional module load in the requester page.

4. User (admin) pops up the services configuration pane

  1. The configuration panel JSP checks the available optional services and their load status
  2. The configuration pane loads a widget for every available service and presents it to the user along with the current service state (activated or not) and the current configuration data.
  3. The user selects or unselects the required services and manages the local configuration settings as required
  4. Once the user confirms their choices, the settings are stored, the services pane closes and the application reloads with the new services/settings enacted
  5. If the user cancels the configuration pane at any stage, the settings remain as they were before
  6. User is presented with a long for each service to reset the local configuration settings for that service - if this is used, the service upon reload uses the app wide settings.
