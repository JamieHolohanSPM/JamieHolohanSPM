Background

Selenium is a portable software testing framework for web applications. Selenium provides a record/playback tool for authoring tests without learning a test scripting language (Selenium IDE). It also provides a test domain-specific language Selenese to write tests in a number of popular programming languages, including Java, C#, Groovy, Perl, PHP, Python and Ruby. The tests can then be run against most modern web browsers. Selenium deploys on Windows, Linux, and Macintosh platforms. It is open-source software, released under the Apache 2.0 license, and can be downloaded and used without charge.

The test automation team within Curam have begun to use Selenium as the functional automation tool of choice. To date, a wrapper infrastructure has been created to allow such tests to be written more easily by developers. It is now proposed to release this infrastructure.

However, a stumbling block to the release and the maintenance of such tests is the lack of a unique identifier in many of the core HTML elements generated for the Curam user interface. This has led to issues with regards to identifying elements to click on and populate within the test scripts and more importantly the fact that these scripts need to work from product release to release (something which is currently not guaranteed as there is no HTML contract with a customer and the HTML structure may change from release to release).

A solution to this is to generate a unique identifier for the HTML elements that this testing framework is interested in and which will guarantee to be the same in future releases. This will allow developers to write the automated tests once with the guarantee that the next product release will not break those tests (at least from a unique identifier generation point of view). The fact that these HTML elements will have a unique identifier will also make the writing of these tests easier and more succinct with an added advantage that the framework to support this may be refactored to take this new feature into account.

NFR Note                                     
Accessibility The ability of the system to be used by people with disabilities. When new technologies are being used for the UI very careful consideration should be given to whether screen readers can be used with the technologies. This applies specifically to usage of JavaScript, Flex and SVG. The use of colors to provide functionality should be considered (and avoided), along with resolutions that will be used by vision impaired users.
Availability & Resilience The ability of the system to be fully or partly operational as and when required and to effectively handle failures that could affect system availability. This should be considered when working outside of the normal infrastructure and deployment. For example, the XML Server is not deployed in an application server and has special handling for fail-over and load balancing. Careful consideration should be taken before adding new components, e.g. EJB Beans, MDBs, to the infrastructure.
Development Resource The ability of the system to be designed, built, deployed and operated within known constraints of people, budget, time, etc. In general there is not much information required here, however where special skill sets or people with siloed knowledge are required to work on the change/enhancement, this should be detailed here.
Evolution The ability of the system to be flexible in the face of the inevitable change that all systems experience after deployment, balanced against the costs of providing such flexibility. Consideration should be given to future proofing the change, including considering how to upgrade to any changes or enhancements provided in the future. Specifically designs should accommodate elements such as localisation (text, dates, etc.), customization, configurability, and componentization. Providing these features in future releases, when not considered up front, has caused impacts for customers in the past and has prevented us from providing the features in some cases.
Internationalisation The ability of the system to be independent from any particular language, country or cultural group. Localisation and internationalisation are a large concern for Cúram. Support should always be provided to view information in multiple locales, based on the user and server locale, along with support for working in multiple time zones.
Location The ability of the system to overcome problems brought about by the absolute location of its elements and the distances between them. For the most part this is not a major concern, however we should never design based on the assumption that the customer will use a default configuration
Performance and Scalability The ability of the system to predictably execute within its mandated performance profile and to handle increased processing volumes. Performance and scalability are a concern and should always be considered during design. Where necessary, the Performance and Benchmarking teaming should be engaged early to help with determining impact of potential changes.
Regulation The ability of the system to conform to locale and international laws, quasi-legal regulations, company policies and other rules and standards. Regulation includes the usages of third party tools and the licensing issues associated with these. New jars should never be introduced without considering the legal implications and licensing of the jars. This includes usage of GPL licensed code. Also, usage of code freely provided on the web and not properly licensed should be avoided.
Security The ability of the system to reliably control, monitor and audit who can perform what actions on what resources and to detect and recover from failures in security mechanisms. This is a concern where new features are added that are not contained within the normal Cúram security.
Usability The ease with which people who interact with the system can work effectively. How the change will affect the user and how they use the application/feature should always be considered. Where necessary the business analysts and our user interface experts should be engaged early to ensure we meet usability requirements. e.g. printing.

 

Design Option Re Attribute to use

Technically, web driver tests do not require the CDEJ to use the "id" attribute.

Option 1 - Use "id" attribute


Pros:-

  1. Semantically this makes the most sense.
  2. Dedicated functions in Selenium to getById.

Cons / Complications:-

  1. Impact
    Curam already generates a number of values into various IDs and we would need to figure out the impact of changing these values. From an FVT perspective they think that they would not be impacted by removing the old IDs.
    Would customers be impacted? If so, do we officially support them on these IDs? Either way, impacting customers would be best avoided.
     
  2. Usage restrictions

    From w3c:-

    HTML defines these naming rules:

    • Must contain at least one character
    • Must not contain any space characters
    • In HTML, all values are case-insensitive

    In HTML5, the id attribute can be used on any HTML element.

    In HTML 4.01, the id attribute cannot be used with: <base>, <head>, <html>, <meta>, <param>, <script>, <style>, and <title>. This shouldn't be a problem for us as these don't concern the testing team.

    Note: HTML 4.01 has greater restrictions on the content of id values (for example; in HTML 4.01 id values cannot start with a number).

    Additionally, a comment in our own HTMLUtils.getUniqueID() states that a "-" is not valid in a HTML ID value. [citation needed]

    From stackoverflow:-

    For HTML 4, the answer is technically:

    ID and NAME tokens must begin with a letter ([A-Za-z]) and may be followed by any number of letters, digits ([0-9]), hyphens ("-"), underscores ("_"), colons (":"), and periods (".").

    HTML 5 is even more permissive, saying only that an id must contain at least one character and may not contain any space characters.

    The id attribute is case sensitive in XHTML [citation needed].

    As a purely practical matter, you may want to avoid certain characters. Periods, colons and '#' have special meaning in CSS selectors, so you will have to escape those characters using a backslash in CSS or a double backslash in a selector string passed to jQuery.

    In summary it would be best to begin with a letter and the rest should contain only letters, digits and underscores.

Pros:-

  1. FVT can easily find an element by an xpath expression that uses the new attribute name.  This is effectively as easy as getting an element by id once a central function is added.

Cons:-

Design Option Re Format of the generated ID

Potential building blocks for the unique identifier of any element:-


<Page ID>

<Cluster / List Div ID>

<ListRowBeginTag><index R-1><ListRowSeparatorTag><index R><ListRowBeginTag>

<ListColBeginTag><index C><ListColEndTag>

<Field ID>
 

1. page id

- need to figure out what to provide for this in IEG, PagePlayer and anything else I haven't thought of where page id may not be known

2. cluster / list div id

3. list row index

- This is already available and it's unclear whether anything we can provide around this will be helpful as list rows are dynamic based on the data and sort orders and filters.

- Would it be possible to create a row id by combining all the links from an entire row? Usually links from a row item will typically include parameters which uniquely identify the object represented by the row.

4. list col id

- note, this is not the index but rather based on the underlying CONNECT

5. field id

- For UIM pages this might be produced from the underlying CONNECT element. This might get distilled down into the most immutable parts.

 

If we want to minify the new unique ids, all of the above might be put through a hashing algorithm to produce a short unique ID.

 

While FVT could simply use the shortened end result of this, changes in pages and page flows make this a bad idea, i.e. deprecated pages being replaced by new very similar pages. It would be better, therefore, to give FVT the formula to create the long form of these ids and then to hash it before locating the element in question. Updating a test to run with a new page superseding an old one could then be a simple process of updating the page id portion in all the test references rather than inspecting everything on the page again at runtime and grabbing the new hashes.
