Summary

The JavaScript in React needed here to configure the same screen is much neater and compact. In order to enable the use of ReactJS components like
 this instead of using UIM, we would first have to expose the Server Interface Beans through RESTFUL APIs. Service of Engagement (SOE) team have deevloped Rest Infrastructue so we may have to look at taking this on (probably into SDEJ)! See here for info on how to configure REST APIs : https://www.ibm.com/support/knowledgecenter/SS8S5A_7.0.0/com.ibm.curam.content.doc/MSDK/msdk_t_create_simple_rest_api.html
 
 What would be the first steps in replacing UIM with something like React components for UI Widgets?:
 (1) Taking on an enhancing Rest APIS developed by SOE - needs collabaoration with them and possibly WCM
 (2) Look at what features we could potentially build new infrastucture in order to make them more flexible for cusmtomer e.g list (pagianed, multisleect, sortable) and clusters maybe to start.
 (3) Look at updated need to UIM generator to move to this new flexible way of building widgets.
 (4) Decide what parts of applications should take on this new infrstructure.
 
 Impacts from moving from UIM to JavaScript to make widgets more flexible?
 In essence what we would be doing is generating the same HTML (except for Dojo HTML syntax) in the new JavaScript modules so there should be no impacts. However React does produce it's own HTML so would have to ensure there are no impacts to other solutions further up the stack or to customer widgets. Also would have to ensure no impacts on automated testing.
