Background

MSD have the following use case sceanrio. In Citizen Engagement SPM/Citizen Engagement provided the necessary hook points to allow MSD to implement a search ahead feature when entering addresses. The idea is that as you start to type an address, it is searching/listing possible matches so it makes it easier to select the correct address without having to enter in all the address details. Clients are very transient apparently, so MSD process 1000's of changes in address per week. Our process out-of-the-box for caseworkers for handling a change in address is considered quick clicky/clunky in comparison to this search ahead feature they've implemented in Citizen Engagement.
So MSD want to do something similar for the caseworker application - where on entering an address they can do a search ahead that allows caseworkers to select the address without having to enter all the details. However, address is dynamic evidence, which complicates things a bit - it's not just UIM.


MSD wanted us to see if there is any way we can enable them in doing this - similar to what the product does in CE where there is provision for the necessary customization points to allow MSD to do this themselves.


Subsequently, there is a similar request in from Product Consulting (from Podge). There is a large bid in progress at the moment for ESDC (Employment and Social Development in Canada). A demo is being built and they want to incorporate a similar feature - have a type ahead search but this time its searching on Employers when capturing Employment evidence - but its the same issue in that it is a dynamic evidence type so they would need us to enable them (product consulting) in building this in their demo.


Given the two high profile requests around this, there is a technical investigation required to:

(a) understand more what Citizen Engagement did for MSD

(b) consider if there's anything we could do in dynamic evidence to open it up and make similar possible on certain (configured?) fields.

 

Once we understand what potentially could we done, we will be in a better position to have discussions around whether we actually do this, how long that might take, who could do it, what has to drop to get this done!!

 

Requirements

  • Requirement 1: The application shall allow a developer to configure a field which will ensure that when the user types into that field in the SPM application, a search will be performed to retrieve the possible results based on the data the user had input up to that point.
  • Requirement 2: The search needs to be performed in a "dynamic" basis as the user types i.e. analogous to filtering the results being displayed to the user as they enter more data.
  • Requirement 3: The search being performed needs to be configurable per field.
  • Requirement 4: The data being returned from the search needs to be configurable per field.
  • Requirement 5: The configuration shall be applied to a range of data types? (ADDRESS data, STRING data, other types)?
  • Requirement 6: The data returned from the search needs to be populated into further form fields on the page containing the search if the user selects on the entries returned (probably only applicable for multi field data such as addresses or applicable to all searches (I am searching for an employer, if I select an employer, should details of that employer returned be populated in a set of pre-defined fields)?
  • The feature shall be available for both static and dynamic UIM pages and the fields contained within.
  • Others?

 

  • RTC Enhancement 245865 - (RFE) Provide UI capability and integration hook to perform the address search and validation in Curam (UIM, Dynamic Evidence, IEG) .

This enhancement is specific to addresses only. If we were going to add a feature, it may have to be more generic that or we could confine it a set of predefined field types.

Design Options

There are a number of design options listed here, applicable to both dynamic and static UIM pages. These are linked to design questions, the answers which are also listed here for clarity.

Question 1: The Type Ahead feature has to be done by PD or the customer could do it.

Answer 1:

  • Static UIM: Yes. Right now, the customer can achieve this. This has been already completed by Podge for the demo where a JSP Scriplet was added to the required pages which included some JavaScript on the page. The JavaScript contained the capability to perform the required search as the user types and populated the specified fields on the page when a selection was made.
  • Dynamic UIM: No. There is no hook point available for the customer to add in such functionality into a dynamic UIM page.

Question 2: If the customer can do if, do we have to provide anything to enable them (a hookpoint or anything else)

Answer 2:

  • Static UIM: They can do it right now without any hook point
  • Dynamic UIM: There is no way for the customer to do this right now without some infrastructure changes.

Question 3: If PD have to do it - what size/effort is involved here? How invasive is the change? Is it something CST could do or would it be a project?

Answer 3: see below

 

Option 1 - Add support for SCRIPT and/OR JSP_SCRIPLET to Dynamic UIM

We have seen where the customer/lab services have enabled this feature on a static UIM page by their usage of the SCRIPT and JSP_SCRIPLET tags in UIM.

 

<PAGE
  PAGE_ID="CCSIntakeParticipantWizard_addParticipant"
  WINDOW_OPTIONS="width=1025,height=600"
  SCRIPT_FILE="PersonSearchAJAX.js"
>

  <SCRIPT
    ACTION="initAJAX()"
    EVENT="ONLOAD"
  />

 

  <JSP_SCRIPTLET>
   <![CDATA[
    out.println(" <div class=\"cluster cluster-with-header is-uncollapsed label-field\"><div style=\"zoom:1;\"><div class=\"header-wrapper\"><h3 class=\"collapse\" title=\"Search By Name\"><span class=\"collapse-title\">Search By Name</span></h3></span></div></div></div>");
    out.println("<div class=\"cluster cluster-with-no-header label-field\" >");
    out.println("<table> <colgroup><col width=\"18%\"><col width=\"32%\"><col width=\"18%\"><col width=\"32%\"></colgroup>");
    out.println("<tr>");
    out.println("  <th class='label'><span label-'label'>Name Search</th>");
    out.println("  <td align='right' colspan='3' class='codetable last-cell'><input autocomplete=\"off\" id=\"completeTable\" name=\"personSearchAJAX\" type=\"text\"></input></td> ");
    out.println("</tr>");
    out.println("</table> ");
    out.println("</div>");
   ]]>
  </JSP_SCRIPTLET>

 

We can see the script file is included and then the function to be called when the page loads. The JavaScript included then takes care of all the processing required to be performed when the user types in the specified field.

These UIM elements are not supported in dynamic UIM (https://www.ibm.com/support/knowledgecenter/en/SS8S5A_7.0.6/com.ibm.curam.content.doc/WebClientReference/r_WEBCREF_Unsupported1Page1.html).

The idea here would be to add support for these elements into dynamic UIM which would allow the customers to add whatever JavaScript they require to the page.

One caveat here is the dynamic nature of the potential field names (especially if it was a requirement to assign field values on the search function return). Some further analysis would have to be done on this.

Pros:

  • Potentially could be a cheap way of doing this.
  • Would allow the customers free reign to perform their required actions (this could actually be viewed as a con!!).

Cons

  • No control as no real infrastructure support for the desired feature. The customer is free to add whatever JavaScript they like to support what they need to do.
  • If customers start to get the JaavScript wrong, could be a maintenance cost.
  • Could be costly to implement "per page" as each page would need the change and potentially new JavaScript to support it (again no real infrastructure support).

ESTIMATE: 1 month

Option 2 - Add support for a new TYPE_AHEAD_SEARCH domian, a new renderer(s) and new configuration

The standard way up to now to ensure that different fields are rendered differently is to provide a new domain with a specified renderer configured or it. Once the CDEJ infrastructure sees that new domian then, it reads the new configuration and renders the desired HTML and associated JavaScript for that field.

In this scenario, a new domain definition would be created in the product and configured to a new renderer. This new renderer would then take the required configuration (what search to invoke, what return data to return, what to do with the return data etc.), all supported in the standard CDEJ renderer framework fashion.

  • New Domain Definition: TYPE_AHEAD_SEARCH
  • New Renderer: TypeAheadSearchRenderer
  • New Configuration:

Each field then where this type ahead search functionality is required would need to have this domain specified.

There is a debate on whether there should be a new domain and renderer per field type or if one generic renderer with some configuration would suffice to build out the use cases we have (Addresses, Employer Name Search etc.

Pros:

  • Standard Infrastructure mechanism for rendering different content and functionality for a particular field type.
  • Would be infrastructure support for allowing these fields to be defined and configured, this would be make it explicit and a first class citizen with regards to what the CDEJ supports.
  • Make this a documented supported feature with a mechanism defined as to how customers would implement it.
  • If a facade function was changed to use the new domain, all of the pages using that facade would be updated automatically.

Cons

  • Existing pages would have to be changed to use the new domain. If so, could potentially be very impactful?
  • Would we need a different domain and different renderer to each "type ahead search type field? If there was no generic functionality between the type ahead search fields, could be more expensive to implement.
  • All the pages with the new domain specified would be rendered as type ahead search (i.e. could not be switched off a per page basis)

ESTIMATE: 2 months

Option 3 - Add support for new UIM mark (or reuse existing mark up) to explicitly mark up a field as a Type Ahead Search

The other standard mechanism for allowing fields to "behave" differently is to mark up explicitly that the field should behave differently, providing the required configuration and then allowing the generators to generate the output based on that output.

In UIM there does exists a CONTROL attribute for a FIELD element:

  • https://www.ibm.com/support/knowledgecenter/en/SS8S5A_7.0.6/com.ibm.curam.content.doc/WebClientReference/r_WEBCREF_Reference1Field1.html

This could be augmented to specify a new type called "TYPE_AHEAD_SEARCH" where new content would be generated. The configuration required then could be specified as UIM also supports an associated CONFIG attribute. Again, some research would have to be done on requirements and to determine whether or not, these elements would support it or new ones would be required. Either way, this mechanism would introduce UIM mark up as a mechanism for satisfying the requirements.

 

      <FIELD
        CONTROL="TYPE_AHEAD_SEARCH"

        CONFIG="Employer.Name.Type.Ahead.Search.Config"
        HEIGHT="4"
        LABEL="Field.Label.EmployerName"
      >

Pros:

  • Standard Infrastructure mechanism for rendering different content and functionality for a particular field type. Would be available for static and dynamic UIM pages.
  • Would be infrastructure support for allowing these fields to be defined and configured, this would be make it explicit and a first class citizen with regards to what the CDEJ supports.
  • Make this a documented supported feature with a mechanism defined as to how customers would implement it.
  • Allow individual pages with the same field type to be marked up differently (one field could have type ahead enabled, another field of the same type may not)

Cons

  • There is no support at the moment for the "CONTROL" element in dynamic UIM, so this would have to added first before the additional support is added for the type ahead search feature.
  • Existing pages would have to be changed to use the new mark up. If so, could potentially be expensive if there are many pages to change.
  • Different configuration may be required per search type.
  • New mark up and UIM element/attribute support may be required if the existing available UIM mark up does not suffice.

ESTIMATE: 2-3 months
