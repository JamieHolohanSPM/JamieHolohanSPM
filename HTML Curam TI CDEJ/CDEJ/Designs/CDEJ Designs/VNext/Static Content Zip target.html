Background

The target "zip-static-content" is not fit for purpose. Currently, it is causing the following issues when used in development environments:

  1. Does not have an exclusive option for creating an external client static content zip archive.
  2. Numerous static content files are not being zipped into the created archive from the Static Content directory.
  3. Duplicate static content files are being zipped into the archive and causing issues when the archive is exploded.
  4. There is unclear/missing documentation around the use of the zip-static-content target.
  5. The exact static content extension types to be archived need to be identified to streamline the zip-static-content target. 

Requirements

List of Requirements

  • Requirement 1: A definitive list of static content extensions will be listed and configured in the target so all static content is zipped into the archive.
  • Requirement 2: No duplicate static content instances will be loaded into the zip archive.
  • Requirement 3: A separate target/ target option will be created to archive all static content for the external client.
  • Requirement 4: Review and created documentation which will provide the customer exact steps for the zip-static-content target.

Design Options

This is optional. If there are is more than one design option available and being discussed, then these should be outlined here.

Option 1

Looking at the zip-static-content target, it currently doesn't pull in all static content resources to the zip target as it points to a subset of directories and file types to what's currently supplied in the WebContent directories for either Curam or CitizenPortal. It also pushes two versions of the same file into the same zip archive directory which can cause issues when the zip archive is being exploded by the user. To resolve this two design changes can be made:

  • A list of all the file types currently residing in the Curam WebContent and CitizenPortal WebContent directories are amalgamated into a list. This list is then run through to decide which file types are definitely or maybe static content file types(it is better to err on the side of caution here and include files which can't be identified as not being static content resources) and these file types are included in the static content zip target as file types to be collected by the zip functionality.(Solves requirement 1)
  • The option duplicate="preserve" was set on the zip functionality in the ant target so that if a duplicate file was pushed in to the same archive, the original file would be preserved and the new file discarded rather than having two files in the same zip archive directory.(Solves requirement number 2)
  • Target option is already included in the static-content-zip target to point to other static content resource locations to create a zip target for that static content location. This just needs to be publicised through the target description and the KC documentation.
  • Documentation will be updated to provide a customer with the exact steps to zip the static content for the Curam application and for the external application.

Design User Interface

If the new enhancement contains any User Interface element, then a UX spec may be required. These include screenshots associated with the new feature, usability concerns, styling and any other UI considerations. Designers and UX team should be involved here. The spec should be associated with this design.

Note: New UI features should also be tested to ensure they work on a tablet device. From 7.0.1.0 we state we support tablet devices so we need to make sure we are not regressing in this area.

Key board short cuts should also be considered if developing features that interact with the user depending on mouse clicks/key board interaction (i.e. back browser button, refresh etc.)

Browser differences should be considered for features being developed ensuring that the required browsers are tested and supported.

Is the feature specified for the internal caseworker application or for the external Universal Access application (or both)..

Accessibility for both the desktop application (JAWS and IE11 is the supported combination) and tablet (VoiceOver on an iPAD using the keyboard and gestures is the supported combination) should be considered for new UI features.

Table Support

 

  • As of v7.0.1 Curam will be certified for use on tablet devices. This introduces a new responsibility on Business Analysts, UX Designers and Developers to ensure that they do not change existing or introduce new features that break the Tablet support statement. For the most part this is limited to the production of custom widgets (non-standard UIM) and how they perform when using touch screen controls. Please note the following:
  • Tablet resolution is typically lower than the minimum recommended resolution of 1366x768. This means that the rules around wrapping of text have been relaxed for tablet, but teams should make a reasonable effort to ensure that any wrapping is not excessive and does not affect the ability to understand and use the application.
  • Where custom features are introduced, teams must test using a iOS and Android device to ensure that the feature is compatible with touch screen controls.
  • Where custom features are introduced, they must meet accessibility requirements for mobile devices.

Design Resilience Measures

This is mandatory. Some thought should be given to how resilient the new feature is.

  • What happens if something expected/unexpected goes wrong?
  • What happens to the system?
  • Will the user be able to take corrective action in the system without the need to resort to application logs/SQL searching/data manipulation outside of the system.
  • Will the data be left in a consistent state within the system after the exception is thrown?
  • Will there be messages given to the user? What will these messages be?
  • Will there something written to the logs? What will this be?
  • Does the feature require some of the exception data to be captured and an admin function provided to allow the situation to be re-mediated?

Design Logging

Think about what the logging could be to avoid revealing PI information but giving enough information to allow effective debugging to occur in the event of an unexpected issue? JMX considerations should also be added here.
