Background

(The following is taken from the Word Integration Playback 1 from 2018 https://ibm.ent.box.com/file/355338927256, Castro, Kovanas,McFadden et al)

CÃºram SPM  currently has separate Microsoft Word integration solutions for the Internet Explorer and Chrome browsers on the client desktops:

The current solution allows a MS Word document to be created automatically from a template where the template details are set in SPM before the document is presented 
to the user for editing. 

When the user finishes editing the document and closes MS Word, the document is saved to the IBM CÃºram SPM database.

The current solution has two separate implementations, both of which are supported on Windows only.

  • One for Internet Explorer 
    • Java applet and the COM bridge combination for the browsers which support applets 
  • One for Chrome
    • After Google Chrome announcement (Sep, 2015) they were dropping Java applet support,CÃºram SPM had to build a âChrome specificâ solution using the Native Messaging Bridge 
    • This solution also requires a Java Run-time and is documented here 

Some significant events  have  resulted in the current solution having two separate implementations along with the more pressing need for a new future proof solution. 

  • Following Google's announcement, Microsoft have followed suit and dropped Java plugin support from the Edge browser.
  • As part of JRE 9.0 (Released Sept 2017), Oracle announced the deprecation of the deployment stack, required for Applets and Web Start Applications. These were subsequently removed from JRE 11 (Released Sept 2018)
  • The current applet solution has a high maintenance cost with multiple APARs from customers which prove difficult to reproduce, diagnose and fix due the number of moving parts.

 

Project Timeline to date:

The Word Integration Project has been run in phases

 In the first phase, the Research And Development team investigated possible alternative solutions https://ibm.ent.box.com/folder/49079899522

These technologies were considered

  1. For On-premise
    • a.Microsoft Office Add-Ins;
    • b.Office URI Schemes and WebDav;
    • c.Native Messaging API;
  2. For Cloud
    • a.Office Online 365 with WOPI;
    • b.Office Online 365 with MsGraphand MsOneDrive

The recommendation made in July 2018

For an on-premise solution, R&D recommends WebDAV and Office URI Schemes.

  • This is a direct replacement of existing technology. Please note that R&D did not perform a proof-of-concept using CÃºram SPM caseworker application.
  • The spike was purely technical to prove the technology using a sample web application hosted on a simple python HTTP server integrated with WsgiDAV (python-based WebDAV server) and a stub REST API using stubby4j.
  • The research did not perform any non-functional evaluation of the WebDAV service such as:reliability, performance, concurrency, scalability, security, high availability, fault tolerance.
  • These non-functional evaluations must be performed by the project team. R&D also did not assess multiple WebDAV servers âonly WsgiDAV. A project team must decide which WebDAV servers CÃºram SPM should certify âsimilar to what was done for CMIS.
  • The non-functional scenarios above should be benchmarked for production.

For a hybrid cloud architecture, R&D recommends Office Online 365 with WOPI.

  • R&D did not execute any technical spike on this Cloud alternative due to associated costs, but the research indicates this is a strategic wayto integrate CÃºram SPM with Office on the Cloud. A project team might decide through design thinking sessions if the Cloud approach should be implemented.

For Appeals.Due to low adoption of Word integration with Appeals reports by CÃºram SPM customers (i.e. only one customer), R&D recommends the project team to assess Microsoft Office Add-Ins as the way to solve the Appeals Clause problem

  • MsOffice Add-Ins has been used already in the industry to manage Clauses. However, it is up to discussion if CÃºram SPM should develop the Add-In or the customer. Design thinking may help to decide.

 

In the second phase, a team was staffed to begin work on this project starting in late 2018, to implement a WebDAV solution.In the second quarter of 2019, the project is suspended.

Why was the WebDAV solution initially chosen.

  • WebDAV would provide a browser agnostic solution to reduce maintenance costs for CÃºram.
  • From a customer perspective,the number of moving parts on the client side would be reduced, and the complexity would be moved to the back-end

Characteristics of the WebDAV solution:

  • The editing experience is delivered by the Word client
  • Microsoft Word acts as WebDAV client and can use a WebDAV URI to access a file on a remote WebDAV server. This ensures that the file does not have to be downloaded to the client machine (AD-004 â Technology ). 
  • It has as a prerequisite, a WebDAV Class 2 Server (https://www.webdavsystem.com/server/documentation/ms_office_read_only/)
  • Separate Web DAV Server component that runs as a separate process on the server side.
  • The file is to be persisted to the SPM database (as before)
  • Temporary files are on the WebDAV server. CÃºram EAR will create the Word document from a template, on the WebDAV server. CÃºram EAR will delete the file as part of closing the editing session
  • WebDAV Server has to be able to act as a Reverse Proxy, to trigger the CÃºram EAR to save the file content to the SPM database and update the state of file metadata in the database during the Editing Session.
  • Project identified IBM HTTP Server as being able to act as a web server that provides WebDAV Server Class 2 and reverse proxy support (AD-005 â WebDAV Server Implementation)

Consequences of choosing WebDAV

  • Two or more WebDAV servers needed for high availability and fail-over 
  • This implies that will need to access to shared file-system for fail-over (AD-006 â WebDAV Server File System )
     

In 2019, the following factors caused us to re-consider the decision to implement a WebDAV solution

Factor #1: Technical Issue for Hamburg, our largest sponsor customer.

  • The choice of IBM HTTP Server (IHS) might be problematic. 
  • Hamburg already are using IHS as their web server and it is deployed on z/OS. See link
  • IHS does not support WebDAV enablement on z/OS. See link
  • Consequently, they cannot use IHS on z/OS as WebDAV Server

Possible Solutions for this.

  • A) Hamburg deploy the WebDAV server to a non z/OS machine
    • For fail-over, there will need two or more non z/OS machines.
    • Additional cost and overhead for the customer.
  • (B) Raise a ticket against IHS to add WebDAV support for z/OS
    • Matt Nuttall, WebSphere and IBM HTTP Server on z/OS Support  writes:
    • "Our documentation is correct: WebDAV is not ported to Apache on z/OS and there are no plans to include it in the future."
    • We would have to create a deliverable dependency on another product , who currently have no plans to deliver this support.
  • (C) Implement the missing modules ourselves for z/OS. 
    • This defeats one of the purposes of choosing IHS per "AD-005 â WebDAV Server Implementation"
    • Support from the IHS team? What happens from a support perspective if there is an issue on this IHS server?
    • Additional development effort developing Apache Modules in C. See link  and  link.
  • (D) Use a Third Party Java Library to deliver WebDAV Class 2 capabilities through the CÃºram EAR
    • The CÃºram front-end becomes a WebDAV server by using a set of Java libraries.
    • This means the WAS server running on z/OS is a WebDAV server. There is no need for a separate process. It does not need to behave as a reverse proxy.
    • However, most libraries that provide support Class 2 require commercial licenses.

Factor #2: SPM Cloud Strategy

Immediate Effects

  • This more relates to what the WebDAV solution won't do, and how it affects deploying SPM to the Cloud.
  • There is a question of how our chosen solution will effect how the SPM product can evolve.
  • For Example: See the previously mentioned requirement for shared storage AD-006 â WebDAV Server File System 

Future Effects

  • While there are some Cloud Solutions that leverage WebDAV , the proposed WebDAV approach retains word processing support as part of the core functionality that SPM is seen to provide natively.
  • This contrasts with industry trend for Cloud where the move is toward delegate the responsibility to a Cloud service.
  • We may have to launch another Word Integration project again within two to five years, to address Cloud requirements as part of a Cloud initiative.
  • For example: If we have customers who also have enterprise Office 365 licenses that follow the direction set by Microsoft, we could have a situation where those customer will want to dispense with Word Client on their machines and go with an Online Editor OR they might want support for both, which is the experience they get with http://office.com/

Factor #3: Launch of Edge Chromium

The following statement written in June 2018, in an R&D document, outlined the difficulty in extending the exiting Chrome solution to Edge

  • The main advantage of using the Native Messaging API, is that itâs the recommended industry standard on the replacement of java appletsâ¦

    it would seem to be a small task to develop a similar solution to work for the Microsoft Edge browser.

     Unfortunately, the Edge browser extensions use the Native Messaging technology in a different manner.

    However, this last statement applies to EdgeHTML. 

Edge Chromium GA'ed on January 15th 2020 and is built on the same foundation (Chromium) as the Chrome browser

The documentation on the Microsoft website indicated that the Native Messaging API implementation is essentially the same as Google Chromes , but there are some minor differences
https://docs.microsoft.com/en-us/microsoft-edge/extensions-chromium/developer-guide/native-messaging

Together with the existing proposal to drop IE 11 support in favor of support for Edge , (for reasons outside of the scope of the Word project) , this announcement makes the Native Messaging API solution more appealing. The thinking was that with minimum development effort the existing solution could be adapted to work on another browser.

 

The question we were left with at the end of 2020 was, if the development and maintenance cost involved for our WebDAV solution, was warranted if it had a short shelf life?

Change of Direction / Phase 3

At a meeting in January 2020 with senior management, the tentative decision arrived at was , that  the development and maintenance cost for WebDAV was not warranted at this time.That the longer term solution would involve looking at Cloud based solutions, and that a shorter term strategy for solving the most immediate challenges for the Word Integration Project, which is the Oracle licensing issue.

For the applet based solution to work, an Oracle JRE v7.x or v8.x must be present on the caseworker client machine. See link.

If customers are using Oracle JRE v8.0 on the client machine, they will have to pay a license.We investigated OpenJDK v8.0 , which does not require a license fee,,but, it  does not provide a Java Plugin for IE11 or Edge.

With WebDAV shelved, we have to look again at the alternatives to WebDAV , a list of the alternatives  that were investigated are listed in the section Design Options. TO-DO List the other options considered.

Most of them would have comparable development effort save for one, Native Messaging API, the current solution supported for Google Chrome.

The Native Messaging API solution also requires a Java run-time but does not depend on applets and can be an OpenJDK JRE.( which doesn't have a license fee)

Our proposal is that we improve on the Native Messaging API solution by extending the solution developed for Chrome to other supported browsers that support the Native Messaging API (Edge or Firefox) AND to improve the robustness of the Native Messaging solution

In February 2020, we developed a patched version of the existing Chrome Solution solution that allows it to work on Edge Chromium. Details of our changes are documented in the section Implementation Notes

At the same time, Stephen Horgan fixed an issue in the Chrome solution relating to the handling of an empty details file, and also made improvements to the logging of the Native Messaging Host. See Implementation Notes .

In February 2020, Feargal Conneelly completed internal FVT testing of thow patch, and in March 2020, a TestFix was provided to Hamburg that included the changes needed to enable Edge Chromium support and Stephen's fixes

Architecture/Design Context Diagram

The diagram below shows the sequence of interactions between browser and front-end and browser extension and the Native Messaging Host, up until the point that editing session begins.It does not show the part of the exchange that happens on Committing/Saving changes made during an editing session.

 

The Word Editing UIM/VIM pages of the SPM application , when accessed by a Chromium based browser will hit a JSP file called file-edit-chrome.jsp.

This JSP will output the JS that executes the file download and subsequently calls the browser extension to start the exchange with the Native Messaging House

The JS that executes in the browser issues three HTTP requests in sequence.

  1. The first HTTP request initializes session data that is used to store the Details File and Content File for a given session
    1. The Content File is a the template for a Word Communication
    2. The Details File is the file that has the substitution values that the Native Messaging Host Java application will uses to do a token replace , filter copy to generate a new editable file on the client machine
  2. The second HTTP requests downloads the Details files to a download location beneath the browser file system
    • The download code uses this Chrome API https://developer.chrome.com/apps/fileSystem  to persist the download file to these locations)
  3. The third HTTP request downloads the Contents file 

It is after the download of the file that the message exchange with the Native Messaging Host begins.

The protocol of the Native Message API message exchange in general is described here https://developer.chrome.com/apps/nativeMessaging

Chrome starts each native messaging host in a separate process and communicates with it using standard input (stdin) and standard output (stdout).

The same format is used to send messages in both directions: each message is serialized using JSON, UTF-8 encoded and is preceded with 32-bit message length in native byte order.

The maximum size of a single message from the native messaging host is 1 MB, mainly to protect Chrome from misbehaving native applications. The maximum size of the message sent to the native messaging host is 4 GB.

The sequence of messages exchanged between the CÃºram SPM File Native Messaging Bridge browser extension and Native Messaging Host (NMH) follows this pattern

  • On receipt of a message from the browser extension, the NMH performs an action based on the message, either a set-up check, launch Word , or to quit Word.
  • On receipt of messages from the NMH, the extension will update the pop-up modal that appears when launching a Word editing session and subsequently responds to the NMH with the next message in the protocol sequence

The sequence of messages between the SPM Browser Extension and the SPM Native Messaging Host (NMH) is as follows

  1. The first message is sent by the extension to the NMH, is a JSON Object string with a msg attribute,  whose value is a  '|' delimited string.
    • These values will subsequently be used to indicate to the NMH, where the download files are in the Chromium File System
  2. The NMH, on receipt of this message , responds by sending back a JSON Object string with an attribute text with a value PARAMS_READY
  3. This signals to the Browser Extension to send on the next message. Again, this a JSON Object String with an attribute msg whose value is a "\@\" separated list of values which the NMH uses when communicating progress back to the Browser.
    • The main change that we made for the patch provided to Hamburg was to patch the extension so that during the message exchange it included one addition piece of information which is an indicator of which browser is being used, Chrome or Edge Chromium
    • The oval shown on the right hand side is pointing to where in the message exchange this indicator will be included.
  4. On receipt of this message, the NMH responds by sending back a JSON Object string, with an attribute text with a value MESSAGES_READY
  5. This signals to the Browser Extension to send on the next message. Again, this a JSON Object String with an attribute msg whose value is to CHK_JACOB
    • This indicates to the NMH to check if the required JACOB DLLs, needed for Java-COM integration  are available.
  6. If this is successful, the NMH responds with a JSON Object string, that has an attribute text with a value of JACOB_READY
  7. This signals to the Browser Extension to send on the next message. Again this a JSON Object String with an attribute msg whose value is to CHK_WORD
    • The NMH then checks the Word installation on the client machine. On a successful check,  It sends back a series of 4 messages (using the message string sent to it earlier in the exchange) to indicate the progress of the check.
    • The last of these four messages is a JSON Object String with a text attribute of WORD_READY.
  8. The browser extension will then respond,with JSON Object string, that has an attribute msg with a value of ALL_READY
    • The NMH will perform the filtered copy based on the Content and Details files, to produce a new editable file and then launches the Word application,to edit this new file
    • While doing this, the NMH sends four messages reporting the progress to the Browser Extension, and concludes by sending with a JSON Object string, that has an attribute text with a value of SESSION_STARTED.

 /**
   * Finds the name of the most recently used Chrome/Chromium profile.
   *
   * @param userDataDir the location of the Chrome/Chromium User Data folder
   * @return the most recently update Chrome/Chromium profile
   */
  private static String getProfileName(final String userDataDir) {

    final List<Path> profiles = new ArrayList<Path>();
    final File userDataDirFileObject = new File(userDataDir);
    final File[] profileFiles =
      userDataDirFileObject.listFiles(kChromiumBasedBrowserProfileMatcher);

    if (profileFiles == null || profileFiles.length == 0) {
      return null;
    }

    for (final File profile : profileFiles) {
      profiles.add(profile.toPath());
    }
    // sort in descending order based on last modification time
    Collections.sort(profiles, new Comparator<Path>() {

      @Override
      public int compare(final Path p1, final Path p2) {

        try {
          return Files.getLastModifiedTime(p2).compareTo(
            Files.getLastModifiedTime(p1));
        } catch (final IOException e) {
          return 0;
        }
      }
    });

    if (profiles.isEmpty()) {
      return null;
    }

    final Path mostRecentProfile = profiles.get(0);
    return mostRecentProfile.getFileName().toString();
  }
 

private static final FileFilter kChromiumBasedBrowserProfileMatcher =
    new FileFilter() {

      @Override
      public boolean accept(final File file) {

        return file.isDirectory()
          && file.getName().matches(
            FileEditConstants.kProfileNamePatternString);
      }

    };
