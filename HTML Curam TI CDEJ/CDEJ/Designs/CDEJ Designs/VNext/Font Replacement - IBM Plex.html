Background

This originates from RTC Enhancement Request 234623 which was then copied to RTC Story 248182 to track the work. Most of this section is copied from these.
 

SPM will no longer be licensed to ship with Helvetica from (May?) 2020. As a result, SPM needs to move another font and IBM Design (dkuehn@us.ibm.com) has selected the IBM Plex Font.

For more links and information, please find below:

  • https://www.ibm.com/plex/
  • https://github.com/IBM/plex
  • http://www.carbondesignsystem.com/style/typography/overview

SPM Analysis:

The fonts currently used in SPM are located here: "\TI\client\CoreInf\CuramCDEJ\lib\curam\web\themes\curam\fonts". They include the following:

  • bold
  • light
  • medium
  • roman

These fonts are then referenced in the CSS file which is used for inclusion into the SPM pages:

  • helvetica-neue-ibm.css

There are then three CSS variables files which are used for defining the CSS variables which are used across the application (TI\client\Generator\resources\css)

  • css-variables.properties
  • external-infrastructure-css.properties
  • internal-infrastructure-css.properties

In the "internal-infrastructure-css.properties" file, the font families have been defined:

  1. FONT FAMILIES infrastructure_helv-medium_font-family = HelvNeueMediumforIBM, Helvetica, Arial
    infrastructure_helv-roman_font-family = HelvNeueRomanforIBM, Helvetica, Arial
    infrastructure_helv-bold_font-family = HelvNeueBoldforIBM, Helvetica, Arial
    infrastructure_helv-light_font-family = HelvNeueLightforIBM, Helvetica, Arial

There are multiple references to these variable names in the css-variables.properties file.

Unfortunately, the term "helv" has been used in these names so this may mean that we need to change these variable names and assess the impact across all of the higher stacks (i.e. rename the variables to something more generic agnostic of the font family being used and update all references). This may consume the most time as part of this exercise.

Accessibility

The ability of the system to be used by people with disabilities. When new technologies are being used for the UI very careful consideration should be given to whether screen readers can be used with the technologies. This applies specifically to usage of JavaScript, Flex and SVG. The use of colors to provide functionality should be considered (and avoided), along with resolutions that will be used by vision impaired users.

Both desktop application and tablet devices should be considered in this effort.

  
Availability & Resilience The ability of the system to be fully or partly operational as and when required and to effectively handle failures that could affect system availability. This should be considered when working outside of the normal infrastructure and deployment. For example, the XML Server is not deployed in an application server and has special handling for fail-over and load balancing. Careful consideration should be taken before adding new components, e.g. EJB Beans, MDBs, to the infrastructure.  
Development Resource The ability of the system to be designed, built, deployed and operated within known constraints of people, budget, time, etc. In general there is not much information required here, however where special skill sets or people with siloed knowledge are required to work on the change/enhancement, this should be detailed here.  
Evolution The ability of the system to be flexible in the face of the inevitable change that all systems experience after deployment, balanced against the costs of providing such flexibility. Consideration should be given to future proofing the change, including considering how to upgrade to any changes or enhancements provided in the future. Specifically designs should accommodate elements such as localisation (text, dates, etc.), customization, configurability, and componentization. Providing these features in future releases, when not considered up front, has caused impacts for customers in the past and has prevented us from providing the features in some cases.  
Internationalisation The ability of the system to be independent from any particular language, country or cultural group. Localisation and internationalisation are a large concern for Cúram. Support should always be provided to view information in multiple locales, based on the user and server locale, along with support for working in multiple time zones.  
Location The ability of the system to overcome problems brought about by the absolute location of its elements and the distances between them. For the most part this is not a major concern, however we should never design based on the assumption that the customer will use a default configuration  
Performance and Scalability The ability of the system to predictably execute within its mandated performance profile and to handle increased processing volumes. Performance and scalability are a concern and should always be considered during design. Where necessary, the Performance and Benchmarking teaming should be engaged early to help with determining impact of potential changes.  
Regulation The ability of the system to conform to locale and international laws, quasi-legal regulations, company policies and other rules and standards. Regulation includes the usages of third party tools and the licensing issues associated with these. New jars should never be introduced without considering the legal implications and licensing of the jars. This includes usage of GPL licensed code. Also, usage of code freely provided on the web and not properly licensed should be avoided.  
Security The ability of the system to reliably control, monitor and audit who can perform what actions on what resources and to detect and recover from failures in security mechanisms. This is a concern where new features are added that are not contained within the normal Cúram security.  
Usability The ease with which people who interact with the system can work effectively. How the change will affect the user and how they use the application/feature should always be considered. Where necessary the business analysts and our user interface experts should be engaged early to ensure we meet usability requirements. e.g. printing.

Option 1

Approach: 

  1. Delete the following  4 helvetica font directories from TI CDEJ.

    • /CuramCDEJ/lib/curam/web/themes/curam/fonts/bold

    • /CuramCDEJ/lib/curam/web/themes/curam/fonts/light

    • /CuramCDEJ/lib/curam/web/themes/curam/fonts/medium

    • /CuramCDEJ/lib/curam/web/themes/curam/fonts/roman

  2. Delete 4 files from TI XMLServer (as these are not shipped)

    • TI/BuilderScripts/xmlserver-rtl/fonts/Helvetica-Bold.afm

    • TI/BuilderScripts/xmlserver-rtl/fonts/Helvetica-BoldOblique.afm

    • TI/BuilderScripts/xmlserver-rtl/fonts/Helvetica.afm

  4. Scan all source codes for Helv* files.

Option 2 

  • Introduce Plex font: Replace Helv* font family with the recommended one (IBM Plex)
    Note: we keep "helv"/""helvetica" in the existing style names but point to the Plex font

 

  • Approach: - Includes the Approach from Option 1, do the following:
    1. Add 1 large Plex Sans font file (containing everything we will use from Plex) to TI CDEJ
    2. Update (without renaming) 4 helv font-faces which specify weight. And create an 'IBM Plex Sans font face. Both pointing towards the new Plex font files.
      • For Example: 
        Before After
        @font-face {
          font-family: 'HelvNeueLightforIBM';
          src: url('./light/helvetica-neue-light.eot?') format('eot'), url('./light/helvetica-neue-light.woff2') format('woff2'), url('./light/helvetica-neue-light.woff') format('woff'), url('./light/helvetica-neue-light.ttf') format('truetype');
        }

        @font-face {
          font-family: 'HelvNeueLightforIBM';
          src: url(USE_PLEX_RECOMENED_STRUCTURE_HERE);
        }

        @font-face {
          font-family: 'IBMPlexSansLight';
          src: url(USE_PLEX_RECOMENED_STRUCTURE_HERE)
        }
    3. Update (without renaming) 4 helv font-family properties to 4 variations of the new list
      1. From:

        1. infrastructure_helv-light_font-family = HelvNeueLightforIBM, Helvetica, Arial

        2. infrastructure_helv-bold_font-family = HelvNeueBoldforIBM, Helvetica, Arial

        3. infrastructure_helv-roman_font-family = HelvNeueRomanforIBM, Helvetica, Arial

        4. infrastructure_helv-medium_font-family = HelvNeueMediumforIBM, Helvetica, Arial

      2. To: 

        1. infrastructure_helv-[weight]_font-family = ‘IBM Plex Sans [weight]’, ‘Helvetica Neue’, -apple-system, BlinkMacSystemFont, ‘Segoe UI’, Roboto, ‘Droid Sans’, Helvetica, Arial, sans-serif;

    4. Throughout all source control (all streams contributing to a release), replace all decentralised "font" definitions whose list of font's start with helv (will not be directly after the colon though)
      • example (from HCR/EJBServer/components/HCROnline/HomePages/css/protohomepage.css)

        .hcrHomePage .hcrHomePageTop .hcrHomePageHeadingTitleOne,
        .hcrHomePage .hcrHomePageTop .hcrHomePageHeadingTitleTwo {
            font: 36px Helvetica, Arial, Tahoma, sans-serif;
            color: #222;
            display: block;
            margin-left: 60px;
        }
      • Correct the CSS to:

        .hcrHomePage .hcrHomePageTop .hcrHomePageHeadingTitleOne,
        .hcrHomePage .hcrHomePageTop .hcrHomePageHeadingTitleTwo {
            font: 36px 'IBM Plex Sans', Helvetica, Arial, Tahoma, sans-serif;
            color: #222;
            display: block;
            margin-left: 60px;
        }
         
    5. Throughout all source control (all streams contributing to a release), replace all decentralised "font-family" definitions that start with any type of helv. Choose which family based on the font-weight of elements using the font-family being replaced.
      1. Example (from components/AssessmentPlanning/css/AssessmentPlanning_custom_style.css)
        • Before:
          div#curam_smartercare_ipsum_dijit_charting_graph_InnerGraph_0 > div div {      
              font-family: HelvNeueMediumforIBM !important; 
              font-size: 12px !important;
              color: #464646;    
          } 
        • After:
          div#curam_smartercare_ipsum_dijit_charting_graph_InnerGraph_0 > div div {      
              font-family: 'IBM Plex Sans' !important; 
              font-size: 12px !important;
              color: #464646;    
          } 
    6. Throughout all source control (all streams contributing to a release),  manually sanity check any remaining references to helv that are in neither the font nor the font-family regex pattern.

Note:

  • If a font family represents like this Font-family: {Helvetica, Arial, Tahoma, Sans-serif} then to see 'Helvetica' font the user needs to have the it installed on the machine. The font files we provide are only used when loaded with the variables like 'HelvNeueLightforIBM'.
  • On the question of are the properties external?
    • No they are not. Confirmed with Barry O'Sullivan: our supported customisation path at the moment is strictly though CSS. The use of the references to the CSS properties in their custom CSS is not something we support or have documented anywhere. Now that's not to say they might not have done it as they can see these files in the source code and might  but the key point is that is not supported at all at the moment.
    • the default CSS files are the ones that we have in the Out Of The Box components. We only want them to customize the ones in their custom component(s). Otherwise if they do update default they would break the upgrade path i.e future installations of upgrades would overrdie their customization in out of the box components but if they have them isolated in their custom component they won't get overwritten.
    • Referenced knowledge center link: https://www.ibm.com/support/knowledgecenter/SS8S5A_7.0.7/com.ibm.curam.content.doc/WebClientReference/r_WEBCREF_Development1CascadingStylesheets1.html

Right to Left

Existing Helvetica font files do not contain RTL characters. Running the font files through an online font character support tool reveals that all right to left characters are not supported in our existing helvetica font files

Testing of an Arabic build reveals that the build is falling back to Arial font characters, not Helvetica. 

RTL characters are separated into their own font files, Arabic has its  own font files, Thai ect. The font weights are also seperated into their own font files. 

In our  existing set up, we use 4 font-faces, 4 font properties for light, regular, medium, bold.  

These font-faces point to the font files for the respective weight so to support all characters using only these 4 font faces,

Solution Option 1:  include all characters in the font file for each type. Something that is not typically done.

Option 2: Use unicode range in CSS to have two classes with the same name but only serve the font files needed depending on the character sets used.

Browser support for unicode-range (red:NO SUPPORT, light-green:PARTIAL SUPPORT, green:FULLY SUPPORTED):

  

Option 3: The recommended approach from design is to use implementation option 4b and the CSS classes that comes with Plex.
