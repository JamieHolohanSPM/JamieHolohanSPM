Background

In application streams, currently there is no existing functionality to be able to generate JavaDoc for client side projects. JavaDoc can only be generated for server side projects. When application streams are generating a delivery/release zip, there is a JavaDoc target that is run in the createDelivery.xml file, which lives in ADBuildTools. A <component>/doc/api.zip is generated from this target. A similar target for webclient projects is required which will generate an api.zip for external facing client classes.

Requirements

List of Requirements

  • Requirement 1: A new ANT target in the createDelivery.xml, that lives in ADBuildTools, which is run during the process to create delivery zips.
  • Requirement 2: The target shall be run the same way as the current target, which exists for server-side projects.
  • Requirement 3: An api.zip shall be created for the javasource directory of each component in the webclient folder of the delivery zip.
  • Requirement 4: An api.zip shall contain the JavaDoc file, describing the apis which are available to customers.
  • Requirement 5: The ability to include/exclude specific classes from the JavaDoc contained in the api.zip

Table Support

 

  • As of v7.0.1 Curam will be certified for use on tablet devices. This introduces a new responsibility on Business Analysts, UX Designers and Developers to ensure that they do not change existing or introduce new features that break the Tablet support statement. For the most part this is limited to the production of custom widgets (non-standard UIM) and how they perform when using touch screen controls. Please note the following:
  • Tablet resolution is typically lower than the minimum recommended resolution of 1366x768. This means that the rules around wrapping of text have been relaxed for tablet, but teams should make a reasonable effort to ensure that any wrapping is not excessive and does not affect the ability to understand and use the application.
  • Where custom features are introduced, teams must test using a iOS and Android device to ensure that the feature is compatible with touch screen controls.
  • Where custom features are introduced, they must meet accessibility requirements for mobile devices.

Design Resilience Measures

This is mandatory. Some thought should be given to how resilient the new feature is.

  • What happens if something expected/unexpected goes wrong?
  • What happens to the system?
  • Will the user be able to take corrective action in the system without the need to resort to application logs/SQL searching/data manipulation outside of the system.
  • Will the data be left in a consistent state within the system after the exception is thrown?
  • Will there be messages given to the user? What will these messages be?
  • Will there something written to the logs? What will this be?
  • Does the feature require some of the exception data to be captured and an admin function provided to allow the situation to be re-mediated?
