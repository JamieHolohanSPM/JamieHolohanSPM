Background

Previously the Word Integration was running at the client side using applets, we would open a bridge between Cúram and Microsoft Word that would enable the customer to edit the file and save directly from the browser. The problem is that the browsers nowadays are not supporting applets anymore, so this functionality must be updated in order to work.

RTC Tickets:

Docx4j Spike - https://jazz031.hursley.ibm.com:9443/ccm/web/projects/SPM#action=com.ibm.team.workitem.viewWorkItem&id=195964

Requirements

List of Requirements

  • Requirement 1: The Word Integration functionality will not be using applets.
  • Requirement 2: The Word Integration functionality shall run from the server side without any integration with a specific text editor.
  • Requirement 3: The Word Integration shall use the Docx4j API.
  • Requirement 4: The Word Integration shall be done by using a download / upload method.
  • Requirement 5: The Word Integration will support files with .dotx and .docx extensions.
  • Requirement 6: The user shall be able to use its own templates.

Design

The new functionality will be using the Docx4j API and will be done from the server side by using a download / upload mechanism. The user will inform some data in a previous screen which will then be populated automatically into the template selected (by translating the placeholders from the template into the information provided) and a .docx file will be created and saved in the database. The user will then be presented with a Download button that will enable to download a hard copy into the disk of the populated file. There the user will be able to make changes according to what he needs and upload the updated file by using the upload document feature.

 

The new Word API will contain the following methods:

  • listProperties -> This method will be responsible for listing the properties / placeholders that are present in the template.
  • generateDocument -> This method will receive a list of information and the template selected and will generate a new .docx file using these inputs.

 

Few things to be considered:

  • There won't be any integration between Cúram and any text editor.
  • The application assumes that the placeholders will have the same ID as the fields from the screen used to gather the user information.
  • The user must have a text editor compatible with .docx installed locally.

Design Specifics

 

SDEJ:

  • A new API was introduced in SDEJ that will be used to execute the tasks related to MS Word Document Generation and Population on the server side by using DOCX4J V3.3.5.
  • The methods available in this API are:
    • ​Map<String, String> getTemplateProperties(ByteArrayInputStream templateStream)
      • ​Receives a MS Word Template in the format of an ByteArrayInputStream and iterates over the document populating a MAP with the properties available from the template.
    • ​​ByteArrayOutputStream generateDocument(ByteArrayInputStream templateStream, Map<String, String> properties)
      • ​Receives a MS Word Template in the format of a ByteArrayInputStream and a Map with the properties that needs to be populated in the document and returns a populated MS Word Document in the format of a ByteArrayOutputStream.
    • ​ByteArrayOutputStream generateDocument(File file, Map<String, String> properties)
      • ​Receives a MS Word Template in the format of a File and a Map with the properties that needs to be populated in the document and returns a populated MS Word Document in the format of a ByteArrayOutputStream.

Design User Interface

The interaction between the user and the application in regards of MS Word Document Generation will change from what it was used to be. The document will now be generated on the server side and will be available for the user in a Download / Upload structure. Details of the flow can be found in Future of Word integration in SPM.ppt

Design Validations

Whenever the user decides to use the following method signature:

  • ​ByteArrayOutputStream generateDocument(File file, Map<String, String> properties)

A validation will be done to make sure that the File being sent as a parameter is of a ".dotx" or ".docx" format. (These are the only formats accepted by DOCX4J)

 

 

Design Resilience Measures

 

  • What happens if something expected/unexpected goes wrong?
    • An AppException is thrown and a message will be displayed to the user.
  • What happens to the system?
    • The application will keep running normally but that piece of functionality won't be executed.
  • Will the user be able to take corrective action in the system without the need to resort to application logs/SQL searching/data manipulation outside of the system.
    • Yes.
  • Will the data be left in a consistent state within the system after the exception is thrown?
    • Yes.
  • Will there be messages given to the user? What will these messages be?
    • Yes, the application will have 3 different messages for different scenarios:

      • The MS Word Generator could not save the document.

      •  

        The MS Word Generator could not retrieve the specified template.

      •  

        The MS Word Generator does not support the provided file type.

  • Will there something written to the logs? What will this be?
    • No.
  • Does the feature require some of the exception data to be captured and an admin function provided to allow the situation to be re-mediated?
    • No.

Third Party Software and Legal Requirements

  • Docx4j - Version 3.3.5 - https://github.com/plutext/docx4j/releases/tag/docx4j-3.3.5

docx4j is an open source (ASLv2) Java library for creating and manipulating Microsoft Open XML (Word docx, Powerpoint pptx, and Excel xlsx) files.

It is similar to Microsoft's OpenXML SDK, but for Java. docx4j uses JAXB to create the in-memory object representation.

Its emphasis is on power: if the file format supports it, you can do it with docx4j. But first, you'll need to take the time to understand JAXB, and the Open XML file structure

docx4j was created by Plutext Pty Ltd in 2008 - using OpenXML4J for the OPC piece. Plutext still drives the project, but since then docx4j has benefited from contributions from many individuals. 

 

Use Cases

 

Use Case Diagram

 

Use Cases Description

 

Use Case ID 1
Name Retrieve Template Properties
Actor Developer
Description

Retrieves the properties available to be populated from a specific .dotx template / .docx file.
Preconditions

A valid .dotx template / .docx file must be provided.
Flow The developer makes a call to getTemplateProperties method passing a template (WordProcessingMLPackage).
Expected

The method should return a map of the properties available (Map<String, Property> format).

 

Use Case ID 2
Name Generate Document
Actor Application
Description

Generates a .docx file from a .dotx template or .docx file using the docx4j API to populate its placeholders when available. 
Preconditions

A valid .dotx template / .docx file must be provided.
Flow Whenever the user have informed all the required data and selected a template in previous screens and a .docx document needs to be generated, the application will handle this by getting the chosen template and the data, populating the new document and saving it to the database.
Expected

Having a populated .docx file saved in the database.

 

 

Use Case ID 3
Name Edit Document
Actor User, Application
Description

Enables the user to edit information from a .docx document and update it in the database.
Preconditions

The database must contain a valid .docx file to edit.

The Use Case #2 - Generate Document must be available to generate the new version of the .docx file being edited.
Flow The user must choose the document to edit from a list of documents, update the information as required and submit, the application will then use the Use Case #2 - Generate Document to re-generate the document with the new information provided by the user.
Expected The chosen document updated in the database.

 

Use Case ID 4
Name Copy Clause Description to Clipboard
Actor User
Description

Enables the user to select and copy to clipboard the description of the clauses available in the application with the click of a link.
Preconditions

The database must contain at least one clause to be listed.
Flow The user will be presented with a list of available clauses from the database and will be able to view all of its information, including at least the name and description. A copy to clipboard link will be available from each clause of the list so the user can click on it to copy its description to the clipboard.
Expected The description of the chosen clause should be available in the clipboard.

 

Use Case ID 5
Name Download Document
Actor User
Description

Enables the user to download a specific .docx file from a list of documents.
Preconditions

The database must contain at least one document to be listed.
Flow The user will be presented with a list of available documents from the database and a Download button will be available from the row menu in order to download a copy of the .docx file to the user's hard drive.
Expected A copy of the .docx file should be available in the user's hard drive.

 

Use Case ID 6
Name Upload Document
Actor User
Description

Enables the user to upload a new version of an existing .docx file through the User Interface.
Preconditions

The database must contain at least one document to be listed and updated.
Flow The user will be presented with a list of available documents from the database and a Upload button will be available from the row menu, that button will lead to a new page that will allow the user to select the .docx file from its own hard drive and submit to the application, that file will then override the one present in the database.
Expected A new version of the .docx file should be saved in the database.
