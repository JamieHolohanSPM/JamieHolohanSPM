Background

 Enhancement Request 254587: Smart Navigation: Need Special Caution Clients Icon "Special Cautions are a quick mechanism to convey important points about a client. Incorporating Special Cautions into the smart nav search results provides a quick visual mechanism of presenting additional information that a caseworker may need to be aware of before interacting with the client.
Therefore the request is to incorporate special cautions into the search results of Smart Nav." For MSD - Special Caution is extremely important information they need to convey to their caseworkers so incorporating it onto search results is very important and could impact the safety of those users.

MSD want to use Smart Navigator but previously all case navigation was through a customised search where special cautions were highlighted to the case worker and they could request support if the person was in the office. This was considered a critical safety requirement as case workers had previously been attacked. Smart Navigator bypasses this customization.

RFE 239632 provided a hook point between click of a person search and opening the page that would allow MSD to open a modal if there was a special caution against the user and allow code to be executed to provide the same functionality they had in their customised search page. The request to also have the icons update to give an immediate indication there was a special caution against the user was also requested but considered out of scope for the original request though was noted in that design that it was a valid request.


Existing behaviour

Smart Navigator search displays an icon on each person row result. There are existing issues with this raised internally:

1) There are issues with right-to-left (out of scope?).

2) There are accessibility issues as we use the same alt text for different images and the images are conveying additional information not available to a screen reader, such as gender. See Defect 231841
 

Requirements

  1. The icon displayed for result row must be accessible
    1. Alt text is required that is applicable to the icon being displayed (currently all person rows display the hard coded 'Person Image' alt text.
    2. Icons must be visible in high contrast mode.
  2. The icon selected for each person record must be customisable so the icon/alt text can be based on custom logic.
  3. For the OOTB icons
    1. A new icon/alt text should be displayed if the person has a special icon. This should retain the existing icon information plus the special caution: 
      1. Prospect Person with Special Caution
      2. Restricted Person with Special Caution
      3. Male with Special Caution
      4. Female with Special Caution
      5. Gender Not Specified with Special Caution
    2. The existing icons must be made accessible by providing localisable alt text specific to the icon and reviewing the icons under high contrast mode 
      1. Prospect Person
      2. Restricted Person
      3. Male, Female
      4. Gender Not Specified.
    3. Prospect Person, Restricted Person and Gender Not Specified all use the same icon, Images/person_grey_adult.png - they should each use a separate icon so that customers can change this without having to customise the search.
  4. Approach must also verify existing Defect 231841 "Smart Navigator:Accessibility: We should not use just Colour to show differences between Male, Female and Prospect Persons in Smart Navigator search results"

Non Functional Requirements

Question 1: can requirement 3 be achieved without impacting codeline? This was an internal requirement and can maybe be dropped if it is a codeline issue.

Question 2: do we need to address existing Right to Left issues?

Implementation Notes

  • Existing method to return image url: curam.smartnavigator.target.impl.PersonSearchImpl.getConcernRoleImageURL
  • Struct holding row of person results: curam.smartnavigator.facade.struct.SearchPersonDtls
  • Without changing the existing search api used by PersonSearchImpl this will require an additional API call to read SpecialCaution table (on concernroleid). Given it is a key on the concernrole it should be a lightweight read but we should consider providing special caution with an environment toggle given we've had customer concerns/queries over the impact of smart nav on performance given it can fire requests as the user is typing.
  • The alt text must be pulled from a Message file to be localizable
  • While it may be useful to refactor some of the existing code to pull image url and alt text in lockstep, care must be taken not to change the contract of any methods unless they are private as the search class can be extended by customers
  • Alt text for OOTB must include special caution as well as the existing ootb distinctions (see logic in curam.smartnavigator.target.impl.PersonSearchImpl.getConcernRoleImageURL(long, boolean, boolean)
    • Prospect person
    • Restricted person
    • Male
    • Female
    • Gender not known/specified
