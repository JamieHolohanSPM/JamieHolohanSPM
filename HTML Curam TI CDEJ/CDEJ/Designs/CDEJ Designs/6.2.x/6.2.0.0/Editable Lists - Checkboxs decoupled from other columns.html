Editable Lists - Checkboxs decoupled from other columns

Design Page for Editable Lists - Checkboxs decoupled from other columns

Background

This feature has been developed under the hood since 2011 - used in Appeals

Essentially what they currently are  is a checkbox controlled editable list - where the enabling of a checkbox on a list row enabled editable fields within other columns on that row to be editable.

 

In other word the editable values cannot be updated without the checkbox value being set to true.

 

Requirements

The requirement is that these should NOT be a checkbox controlled editable list. In other words the checkbox field should be decoupled from the other editable columns.

 

So what they want essentially is a 'normal' editable list where a boolean (base domani SVR_BOOLEAN) field does not carry any significance.

 

  • 143633 -https://jazz031.hursley.ibm.com:9443/ccm/web/projects/SPM#action=com.ibm.team.workitem.viewWorkItem&id=143633

 

This means that we would effectively have two types of editable list:

(1) Checkbox Controlled Editable Lists

(2) Normal Editable Lists

 

Any other hidden requirements?

Is there a requirement to support editable Lists where FIELD elements with only a target connection are allowed ( 'Creatable list' support)?

Barry check with CGISS & HCR and there is no such requirement so this is beyond the scope of 6.2. In fact in my opinion this is a different feature, this would be a 'creatable list'.

 

Currently a developer would have a create page with  a CLUSTER that had FIELD(s) with only a target connection, and that would be rendered as an input cluster that would allow the user to input the data (creation of business objects within the application).
 
Post 6.2, if we were to implement an enhancement to allow FIELDS have only target connections within a LIST - this would enable users to create lists of business objects within a create page in the application.

 

One for business to consider I think!

Design Consideration

Current Definition of Editable List?

The current definition of an editable list is that the LIST must:

(1) Have a source connection on each FIELD element within the LIST. In other words a FIELD cannot have only a TARGET connection within a LIST.

     NOTE: Since the feature was introduced there has been validations that check that there must be a source connection when a target connection is found.

(2) The first FIELD in editable list should have a base domain of 'SVR_BOOLEAN'.
    NOTE: There was never a validation put in place to check that the base domain of editable field within a list had this domain.

 

Solutions to the problem of decoupling checkbox column from the other editable columns? 

(1)  Force a new validation that the first editable FIELD within a LIST must have domain based on SVR_BOOLEAN.

       OOTB Impact of doing this? Does not seem to be an issue as the first field within editable lists do seem to have domain of SVR_BOOLEAN.

       Customer Impact of doing this? More than likely they do have the editable lists constructed the same way as OOTB pages but if this were not the case it would break their builds so really this is a non runner as implications are too severe!

(2) Identify a check box controlled editable list from a normal editable list.

(i) UIM attribute to indicate editable list

Currently there is no UIM attribute to distinguish editable lists which makes the task pretty difficult. It is not possible to add a UIM attribute at this stage because that would mean customized editable pages would not behave correctly without impact. Perhaps V7, this should be considered - BOS?

 

(ii) Update JSPX Generator and renderer to differentiate between the types of editable list based on the Boolean Fields position within the LIST.

What this means is that within the generator at build time when we encounter a FIELD based on SVR_BOOLEAN we would check it's position.

If it was at the first position, we would set a special flag/parameter that would tell the associated Checkbox renderer to implement it as a checkbox controllable editable list. Otherwise we treat it as normal checkbox field which won't be associated with other columns within the list.

 

      OOTB Impact of doing this? Does not seem to be an issue as the first field within editable lists do seem to have domain of SVR_BOOLEAN.

 

      Customer Impact of doing this?

      More than likely they do have the editable lists constructed the same way as OOTB pages but if they didn't it would mean that they should be aware that they may have to put some logic into the associated facade to check that  the value of the checkbox field is true! (because it had to be set to true before)

               See Application Impact below for more details

Application Impact

This is mandatory. If there is no impact, then this is as simple as saying so.

 

If there is impact, then this may describe the impact in detail and outline the upgrade path. If the feature is related to a CLE, this section can be a link to that CLE (and hence the documentation associated with the CLE will suffice).

 

Choices to deal with potential impact

  • We don't care if they have pages with editable lists and it has a boolean field that is not the first field, as this is not really a supported feature!
  • Communicate the impact that if they have pages with editable lists and it has a boolean field that is not the first field then they may need some logic in the facade (as stated above)
  • Give customer a way to list pages have editable lists and it has a boolean field that is not the first field so that they can still be treated as before i.e checbox controlled editable list. There are two ways to that this could potentially be done. Offically with some new setting curam-config.xml that can be used to list these pages or as part of 'dark Curam' where we have a new ApplicationConfiguration property they can use to list these pages.
