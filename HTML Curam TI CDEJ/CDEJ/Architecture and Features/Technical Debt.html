Technical Debt

Overview

The following details are a result of the technical debt analysis completed for the CDEJ repository. This includes all of the source code present in that repository including the test logic present.

Feature/Item Name

 Description Comments

Items to Remove

Feature/Item Name Description Comments
Word Integration Look at alternatives, replace with Import/Export and associated support.  
Portlets There is very limited knowledge of our support for portlets within the CDEJ team . The original requirement seemed to have been to alow the exposure of Curam PODs in a portlet and say we supported that. There is a document which outlines how to do this (CÃºram Portlet Deployment Guide For WebSphere Portal Server). There is an entry in our supported Pre-reqs for Websphere Portal Server but this is also out of date (newer versions are available). There is some infrastructure present but it is not well known how to use this infrastructure of what it provided  
Legacy Widgets

During the technical debt analysis if the CDEJ functionality, a number of so called legacy widgets were identified. These are related and documented as "Domain Specific Controls" but we feel that some have been superseded and others are just not used anymore. Customer usage is not as well known though.

  1. Decision Assist - could remove straight away.
  2. Flex Charts - analysis but probably remove (renderers).
  3. Generic Flex Widget -
  4. Frameset Widget (Tree Control)
  5. Heat Map
  6. Meeting View
  7. Schedule
  8. Decision Matrix
  9. Gantt Chart
  
Tag Infrastructure Move everything to the renderer framework, remove the request handling infrastructure.. currently maintaining two infrastructures for this.. (AbuDhabi not usng either, using the underlying JS framework for rendering). If we get rid of the legacy widgets - Renderer (60%)   Tag(40%) - no external impact to making this change, just maintenance.....  
CDEJ - Screen Previews There is infrastructure present for generating screen previews which outputs HTML files which can be viewed offline. We think this functionality may have been resurrected for the Atlas project but it is poorly tested and maintained (with most of the knowledge of the logic having left the building). Although there is a standalone function present (build time parameter to generate these), it may not be extensively used and could be a candidate for removal. CADT (Curam Analysis Documentation Tool) uses this and is itself extensively used.
