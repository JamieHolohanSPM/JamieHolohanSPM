1. Overview

This design page will cover the following three investigations

  1. Mapping of Java types to JavaScript types.
  2. Structure of the JSON data.
  3. Meta-data required in JSON.

2. Data Types

2.1 Requirements

Req 1: Support all the fundamental types used in Cúram Java Structs in JavaScript 

All of the Cúram types resolve to these types below:

  •       Boolean.class,
  •       Character.class,
  •       Byte.class,
  •       Short.class,
  •       Integer.class,
  •       Long.class,
  •       Float.class,
  •       Double.class,
  •       String.class,
  •       DateTime.class,
  •       Date.class,
  •       Money.class,
  •       Blob.class,   
  •       FrequencyPattern.class

Req 2: Support codetables values

It won't be sufficient to only send back the codetable code. We will need to pass back both the code and the description.

Req 3: Support raw and formatted data

It should be possible for a developer to get data both raw and formatted.

  • Why do we need raw and formatted data?
    • ​To allow JS developer feed data into Dojo widgets, the widgets will require raw data.
    • To allow JS developer perform operations with numbers.
    • To allow JS developer create Date objects...e.g. they may then want to do browser side sorting on dates.

The raw value is the value that the server returns. The formatted value is the value returned from TextHelpers (which is formatted by Domain Converters).

The raw value for dates may need to be treated differently than we currently do in the JSP world. Currently the raw value is a Curam Date object. We may want to send back the raw value as the EPOCH number or a string value with ISO formatted date. This will allow a JS developer to construct a JS Date object.

The raw value for codetables may need to be treated differently than we currently do in the JSP world. Currently the raw value is the code of the Curam CodeItem object. We may want to send back further information such as the codetable name and entire codetable contents. This will allow us cache the entire codetable and avoid bringing it down in subsequent requests.

2.3 Issues to Resolve

2.3.1 Dates & Times

Should we use EPOCH value or ISO format for dates? Pawd has suggested EPOCH in original design page. But out TextHelpers expect a formatted date string.

Investigation

See here https://w3-connections.ibm.com/wikis/home?lang=en-us#!/wiki/We1458f1078dd_463a_9e7c_c54ac9e1e5a7/page/Time-based%20and%20numeric%20types

Options

1. EPOCH Value

2. ISO Format String

Decision

Use EPOCH value.

2.3.2 Codetable Codes

In the JSP setup, the TextHelper makes a codetable code only available to the JSP page. We then have a custom tag to retrieve the description associated with that code which is translated correctly.

For the JS layer, we will need to provide both the code and the description to the user of the API. The code is needed for sending information back to the server and the description is needed to display the code on the screen.

In the case of a simple display page, sending back the code and the description would be sufficient. But for edit pages, the user of the API will need the entire contents of the codetable to be able to display a list to the user.

Options

There are three options here:

  1. Mark a codetable code that is returned in JSON as being a codetable code, then provide a separate mechanism for the user to request the entire contents of the codetable if they want it.
  2. Add a flag to the request JSON data that indicates the request is for an edit page, the response will then contain the entire codetable.
  3. Always return the entire codetable when a struct contains a code from a codetable.

The first option is not a runner as we do not want to provide a mechanism for the user of the API to request the entire contents of a codetable. This is due to security concerns (this comes from Pawd, follow up for more detailed reasoning?).

Security Concerns: the security concerns here are that we do not currently expose codetables to the browser, in that you can't make a HTTP request specifying a particular codetable and get back the contents of a codetable. Only codetable information that relates to codetables a developer has referenced in UIM is exposed.

If we provided a mechanism to expose codetables for our JS layer, then we would need to introduce a mechanism for restricting which codetables were accessible, just as we have a mechanism for restricting facade methods that a particular user can call. The reason for this is that we wouldn't want all codetables to be accessible, and we would only want certain codetables to be accessible to certain users.

The second option has the benefit of not returning the entire codetable for a page that is only displaying data. Spoke to Susan, but she feels we shouldn't make this distinction for mobile applications of a edit versus a display page.

The third option is to always send back the entire codetable. This is simpler than the second option, in that the user of the API doesn't need to think about whether they using the information for an edit or display page. But it also has the drawback of sending more data overall than the second option.

The issue of data usage can be reduced by caching the entire codetable at the browser. This will work by setting a flag in the request to state that the entire codetable contents is not needed. The server will then only return the entire codetable contents if 

Decision

We will go with the third option of always sending back the entire codetable contents when a response struct contains a code from a codetable.

 

2.3.4 Blob

Recommended approach for transmitting binary data using JSON is to firstly encode it in a base64 String, then to transmit as a JSON string.

  • http://stackoverflow.com/questions/1443158/binary-data-in-json-string-something-better-than-base64
  • http://hustleplay.wordpress.com/2010/01/05/sending-binary-with-json/

This is the format that the TextHelper use. See JavaDoc from converter associated with BLOB data:

Blob values are not usually formatted for display or parsed when entered by a user. However, for internal processing, the string representation of Blob values uses base-64 encoding according to RFC 2045.

2.3.5 String with embedded data Structure

The description from Pawd about this particular one is:

This is a string that has another data structure embedded within it (e.g. XML, another JSON string). We just need to ensure we can handle this.

Unless I am missing something here, we just store this as a JSON String. Nothing special here?

From a quick Google, the only thing that it seems we need to consider is escaping of content (e.g. quotes) to ensure the string is valid JSON. Guessing that our Java-to-JSON converter that we are using on web-tier will do this for us?

http://stackoverflow.com/questions/19703774/ajax-response-generates-error-in-content-typeapplication-json-for-xml-content

http://stackoverflow.com/questions/14318647/will-unusual-characters-in-json-break-the-format

Decision

Store XML data as a JSON String.

2.3.6 Frequency Pattern

This needs some further thought regards what we are doing with this in a mobile context. The frequency pattern type is fed into a widget. We have no plans to provide a frequency pattern for mobile. So it's not clear what the best way forward is here.

Options

1) We could just say we don't support frequency pattern for mobile

2) We could send back the formatted frequency pattern (e.g. "The first Friday of every month at 9am")

3) We could send back both the formatted and raw value (raw value of no use unless we expose logic of Freq. Pattern to customer, or provide a widget)

Decision

For raw data we will send back a both the code and formatted string. For the formatted data we will just return the formatted string.

 

 

3. Input JSON Structure

3.1 Requirements

Req 1: Ability to specify a call to a façade method.

Req 2: Ability to specify multiple input structs; this includes zero, one, or more input structs being specified.

Req 3: The order of input structs must match the order as modelled in server façade method.

Req 4: Ability to specify calls to multiple façade methods, all possibly different, in one single request. Methods can be on same or different facades.

Req 5: Ability to indicate whether data should be returned in raw, formatted, or both; it should be raw default.

Req 6: Ability to specify query options to be applied to result of call to façade method.

Req 7: Option to specify which attributes\columns to sort the data on and whether the sort should be done in ascending or descending. See format specified by Dojo Store API.

Req 8: Option to specify offset to return subset of records from.

Req 9: Option to specify number of records to return.

Req 10: Ability to request codetables associated with a façade method.

Req 11: Ability to request the user locale.

Req 12: Ability to request timezone.

3.2 Use Cases

  • Call a façade method with a list of input structs.
  • Call a façade method with a list of input  structs, specify query options to be applied to results.
  • Call a façade method with a list of input  structs, specify response data should be formatted.
  • Call a façade method with a list of input  structs, specify response should not include list of codetables associated with façade method.
  • Make calls to multiple different façade methods within the one single request.
  • Retrieve all codetables associated with façade method.
  • Retrieve users locale.
  • Retrieve users timezone.

4. Return JSON Structure

4.1 Requirements

General

Req 1: Data from façade method is returned in a raw format by default (i.e. it is not formatted by Cúram Text Helpers).

Req 2: Response should contain data in same structure as return struct from façade method to web-tier.

Req 3: Informational messages should be included.

Req 4: The order of batch requests is preserved in response i.e. the first response will match the first request, second response the second request and so on.

Req 5: Complete codetable information is returned when a response contains raw data and a field is a codetable code.

Req 6: Both the raw and formatted frequency pattern value will be returned regardless of whether raw or formatted is request. This is required as we will not be supplying functionality to convert a frequency pattern code to a human readable string.

Req 7: Allow developers specify custom formatters for data types.

 

Errors

Req 8: Status code should be included to indicate success or failure, and on failure what type of failure.

Req 9: Error messages should be included.

Req 10: It should be possible to get complete stack trace for errors. This should be configurable to allow a reduced level of information to be supplied by default, with an option to provide full information when in “debug mode”.

4.2 Use Cases

  • Server returns response to request that was successfully processed; A single record without a codetable field or a frequency pattern field.
  • Server returns response to request that was successfully processed; A single record with a codetable field.
  • Server returns response to request that was successfully processed; A single record with a frequency pattern field.
  • Server returns response to request that was successfully processed; A list of records.
  • Server returns response to request that was successfully processed, with informational messages associated with response.
  • Server returns response after error has occurred.
  • Server returns formatted data.
  • Server returns raw and formatted data.
  • Server returns response to a batch request, where all requests were successful.
  • Server returns response to a batch request, where some requests were successful and some failed with error
  • Server returns response to a batch request, where some requests were successful and some failed with timeout

5. Curam Struct API

5.1 Requirements

Req 1: Should contain data in same structure as return struct from façade method to web-tier.

Handle lists.

Handle lists of lists.

Req 2: Where Java type maps to a JavaScript type, the JavaScript type should be used.

Req 3: Where an attribute is a codetable code, it should be represented as a Curam JavasScript CodeTableItem object.

The CodeTableItem object should provide access to the code and the associated description.

The CodeTableItem object should provide access to the CodeTable object it belongs to.

The CodeTable object will contain the complete codetable information i.e. all codes, all description.

CodeTable is a Curam JS class.

Should allow you pass in a code and get the description.

Should provide default value.

Req 4: Where an attribute is a frequency pattern, it should be represented as a Curam JavaScript FrequencyPattern object.

Definitely for first version this will be display only.

Req 5: Provide access to error messages and codes.

Req 6: Provide access to informational messages.

5.2 Use Cases

  • Access the attribute of a struct, where the attribute is a JavaScript type.
  • Access the attribute of a struct, where the attribute is a codetable item.
    • Access associated code table.
    • Access associated description.
    • Access parent CodeTable object.
  • Access the attribute of a struct, where the attribute is a frequency pattern.
  • Access code.
  • Access human readable string.
  • Access attribute of struct that is a list.
  • Access attributes of struct after a request for formatted data.
  • Access attributes of struct after a request for both raw and formatted data.
  • Access the error code.
  • Access the any exceptions.
  • Access any information messages.
