JavaScript API Requirements

Generic facade Access

We have an API with a simple RPC-like call() semantics, that takes appropriate parameters, executes given Cúram facade and return the results.

dojo/store style API

This is an implementation of dojo/store/api/Store for use in standard list-based widgets.

Eventually most functionality will be implemented (including sorting, pagination and hierarchical data), except for transaction() which is not supported at Cúram level.

There is also a requirement to enable application developer to modify query object generated by a widget using CuramStore before it is sent to the server. This should be possible directly as part of CuramStore API.

Support connections via Worklight

In addition to direct Cúram connection the API needs to support connecting through a Worklight server.

Grouping Requests

The RPC API allows for making  multiple facade calls in one request. Results are then returned as an array of Cúram structs. This is to allow for network traffic optimization by the API client.

An optional parameter allows for specifying that metadata should be pulled in as part of the same request instead of doing a separate call for it.

Sync/Async

Both RPC and CRUD APIs can be used in asynchronous mode (default) or synchronously.

In async mode methods return immediately, but they return dojo/Promise that will be fulfilled later. In synchronous mode methods return the actual objects, but block until results are received.

Cancellation

Cancellation is supported for asynchronous call only via the standard dojo/Promise cancel() function.

Additionally there has been a requirement that when an existing call is being processed (request in progress) and new call is required the existing request is cancelled automatically. I have requested clarification on the need for this and we will then decide on supporting it.

Automatic error handling, retries, user interaction

- error: by default display dialog automatically, but allow for custom handling
- timeout: display dialog automatically, but allow for custom handling
- retry policy: let client decide what they want to do on failure/timeout, number of retries, etc.
- development mode: more verbose messages, server stack traces, etc.
- load mask: automatically manage UI lock-out when request is under way, allow for API client to specify what part of UI needs to be masked

Metadata

This includes things like Cúram struct data types and structure information, value formatting instructions, information to enable client side sorting and more - yet to be defined.

Sorting

Allows doing server side or client side sort.

Server side sorting is done by passing sorting instructions in standard Dojo-supported format to the CRUDStore`s query() method. For RPC API any sorting support is determined by the associated Cúram facade implementation.

Client side sorting will be enabled by passing sort orders for different fields in metadata. (Is this viable performance-wise?)

Pagination

Specific resultset range can be requested by passing the Dojo standard format instructions into CRUDStore's query() method. This enables the API client to implement pagination, or use standard widget pagination plugins where provided.

General Requirements

Support for nested lists

In order to enhance application code clarity, flexibility and server roundtrip optimization app developer should be able to model a dtls Struct whose list items contain other dtls Structs.

Not providing this feature results in additional modeling requirements and writing complex application code to handle hierarchical data. It also causes additional server calls to be needed.
