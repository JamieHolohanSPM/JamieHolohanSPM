What is "Word Integration"

  • user perspective: open Word doc from Cúram, edit and save without manual download/upload
  • admin perspective: make doc templates and define where to put data, upload to Cúram DB
  • developer perspective: put FILE_EDIT widget in a UIM, connect to a BPO that provides template (doc) and data (xml) from DB

High Level Design

  • we need to automate: download, open in Word, fill in data, let user edit, on save upload back to DB
  • done with Java applet that orchestrates the editing session
  • talks to Cúram app in browser (Java->JavaScript) via LiveConnect API in Java plugin
  • it controls Word and IE via COM or pure JS for other browsers

Workflow

Opening Document

  • user loads UIM screen with FILE_EDIT widget
  • template / or document data and XML data are loaded into page
    • what's the mechanism of pre-populating Word fields? Any Word API used for that? (yes, setDocumentProperties or similar, via COM interface)
  • a separate popup is launched that loads a hand crafted JSP with applet
    • JavaScript is used to pop-up the applet window
    • separate browser window is opened - this is to ensure multiple form submits with every document save/autosave
      • A separate handling is coded into the struts action generated for file edit widget - a submit always ends up in the LOOP (pass to itself) action.
      • I guess this is to leave the word integration control (applet) page aside, just for the interaction - the applet should not be relaunched-reloaded with every form submit.
    • the separate window is also used for the historical reasons, there were considerations to move that to the application context (more controllable environment)
  • applet gets document from page and saves it locally to temporary folder (temp)
  • applet launches Word with the local document
  • applet gets XML data from page and updates fields in document
    • might cause performance issues, see TEC-18217 for an example.
  • applet saves the document back to Cúram
    • this corresponds to the form submit and might take time.

Editing Document

  • user then edits document as usual
  • on save or close the doc is saved back to Cúram
    • the local temporary copy is deleted after user closes the document
    • what about the interrupted session? the temporary copy is left in the folder? (yes, this must be the case, however depending on how it is interupted the user might be prompted to save their changes into the local copy).

Saving Document Back to Cúram

  • applet is hooked into Word save and close actions via COM
  • it'll let Word save on local disk, then grab document and update the page with new content
  • applet will then submit the page and wait for it to finish loading
    • this is the reason applet has to be in separate window, because page is getting repeatedly submitted as part of the editing session
  • only after page finishes loading the system is ready for further editing

Search Functionality

  • in some use cases it is necessary that caseworker can search in Cúram app while editing Word doc
  • this is clearly possible with FILE_EDIT
  • so there is an additional parameter that opens a specified UIM page in another separate popup and provides copy/paste facility into Word

I think the possible interruption scenarios need to be described as well (closing the applet/main app/word document in the process) - how it is handled

Java plugin, security, etc.

  • The applet based solution relies on Java plugin technology, which in turn relies on NPAPI for browser integration.
  • In order to allow the applet to access local resources its JAR must be digitally signed.
  • Overall security of the Java plugin has been bad, resulting in frequent updates and changes to the way it works.
  • Also, the industry is now moving away from NPAPI because its own security problems.

FILE_EDIT widget

  • see CWCRM: Cúram Web Client Reference -> UIM Reference -> for Widgets -> FILE_EDIT
  • params: source and target document, XML data, optionally search UIM page ID
  • implemented by curam.omega3.taglib.widget.FileEditTag class
  • responsibility: Embeds an word integration control in the page and supplies it with hidden fields containing the document and template data used to create or edit file with MS Word.
  • outputs a bunch of JS code and hidden fields with data
  • also a unique ID is created for the editing session, to be used for control applet window identification
  • when page loads, JS opens the control applet popup
  • optionally also search window

Obtaining/renewing the certificates

Since applet is used and it needs access to the local file system, it needs to be digitally signed. In order to do that we need valid certificates obtained from the validation authority.

A certificate is issued for a year and should be renewed annually. However, our applet signatures are also timestamped so there is no need for Customers to obtained the freshly signed version.

However when they take the upgrades, they should receive the applet signed with the certificate which is valid at the time of its first run.

There are two parts to obtaining/renewing the certificates: approval and the technical.

The approval process is covered in TEC-14113, the current renewal (current certs expire in April 2015) JIRA is TEC-17916

For the approval:

  • Make sure there is no license in IBM available already

If not:

  •   approval needs to be obtained to purchase the renewal (as it exceeds the sum allowed for auto approval)
  •   2 code signing certificates are needed, SHA1 and SHA2
  •   they must be obtained from Symantec (all validations of our credentials have been done with this organization).

Technically it means that once the renewed certificates are purchased, they need to be imported into the keystore (there are scripts for that).

The batch files for this importing are again on \\tiny.curamsoftware.com\ProductDevelopment\TechInf\WordIntegration\Java Applet

For the new certificates (not renewals) the process is (or seem to be) more complicated with first generating csr (certificate request) files based on some Base64-encoded value which is downloadable from Symantec first, then those csr files are passed back to Symantec and just then the certificates are obtained from them, which again need to be imported (MV to correct all my misunderstandings here!)

It seems that for the renewal we should still go through all the process though, and fill-in all the data to apply for the certificate (CSR generation first of all).

How to Diagnose Problems

First of all make sure the right applet version is used and there is just one version.

Make sure the Java cache is cleared. This is done in the Java Control Panel

  • Control Panel->Java Control Panel, choose the right one for Oracle JRE.
  • General Tab -> View, lists all the jars currently in cache
  • delete all the WordIntegrationApplet.jar instances found, also commons libraries and jacob jars
  • it might also prove useful to delete the existing certificate if any (Manage Certificates, and delete the one from IBM) before testing new/updated version of the control.

Customer has that step documented, might be an issue in test environment, but worth checking with the Customer as well.

  • phase 1: opening control popup
    • popup blocker might be activated, keep that in mind; Curam should be white-listed for the browser to allow this popup
  • phase 2: detecting JRE, starting applet
    • Oracle JRE is the one used, on our test machines we would also have IBM JREs (separate control panels for them, not to be paid attention)
    • warning if no Java found, but also if wrong version found
    • multiple Java versions might exist on the user system, the latest installed would be used for the applet
    • wrong installation order might cause the problems even though the right JRE is also present on the system
    • Java Control Panel is used to examine the installed JRE versions and see which ones are active
    • we do not support multiple JREs installed
    • also check that the JRE is enabled for the browser(s) - Advanced tab of the same control panel
    • 32/64 JRE versions might co-exist, the one used should match the browser (even though this should be picked up automatically, depends on the installation)
    • Security exeption: the newer JRE versions have more strict requirements for the security (medium level has been removed). This might cause the applet to be blocked with the corresponding security message. To avoid it, open the Java Control Panel, Security tab and add the application site to the security exception list at the bottom of the tab (click "Edit Site List", "Add" in the next dialog and type or paste your site address instead of the 'example.com' prompt. Click "OK", close the console, restart the application if launched and greyed out.
  • phase 3: finding references to relevant windows
    • lots of issues there related to the COM bridge used. We loop through the windows to find the ones we need (main app and control windows?)
    • fresh issue where the loop encounters window not complying to the API used and cannot recognize it; fix is being put through to basically skip such windows.
  • phase 4: opening Word, loading document
    • not many problems here, however the already mentioned recent performance issue when it takes too long to open word doc because of numerous fields to be pre-populated
    • Word doc is already opened but shows nothing until all those fields are properly instantiated, is this the manifestation?
  • phase 5: saving
    • the issues here are related to timing. The saving since it is related to submitting (uploading) the document takes time, so next save or close operation might be prematurely initiated, this causes problems without recent fixes.

Testing

IE9 > in IE8 mode notice

One important thing to remember: the technology is sensitive to the browser/document mode (applies to Internet Explorer). When doing the testing on the older streams, the prerequisite document needs to be consulted to see what browser/document modes are supported.

Another thing is that the usual testing way when the browser/document mode is simply changed in the Developer Tools for the later Internet Explorer versions is not applicable when testing the Word integration - it just sets the mode for the main application window, but the applet window is separate and therefore will use the default mode. This can result in various warnings/alerts.

To set up the right mode correctly, Fiddler should be used. It allows for changing the settings for the requests/responses and in this case the response header needs to be set up to indicate the requirement for IE8 mode. This is done in the "Filters" tab of the Fiddler application. First of all make sure "Use Filters"  option is selected; then, at the end of the list in the "Response Headers" section, select the option "Set Response Header" and type "X-UA-Compatible" for the name and "IE=8" for the value. This will force the browser to use IE8 mode for all the IE windows.

The matrix for the machines with certain browser versions and Word (Office) installations which could be used for testing could be found here.
