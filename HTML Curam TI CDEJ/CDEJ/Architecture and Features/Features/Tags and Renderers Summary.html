Tags and Renderers Summary

Summary of the TI Tags and Renderers

  • CoreInf/src/curam/omega3/taglib --- generic classes, tag support. Used by tags and once the tags are gone, these might be mosty removed as well
    • ClientTagException - a dedicated exception class for reporting different failures when enacting a tag.
    • CuramTagSupport - common tag, request and bean interaction base implementation, further superclass in the inheritance chain.
    • HTMLAttributesTagSupport - helper class to distribute tag related functionality; extracts the standard tag attribute holder fields, getters and setters into the base class. The ancestor for most if not all tags, removable if the tags are gone.
    • HTMLComponent - one method interface to standardize the Curam tag output. Valid as long as there is at least one plain or wrapped tag.
    • HTMLControls - supposed to be a generic helper class but only used in the legacy FileEditTag, therefore should be removed along with that tag, which is not used at the JSP level any longer.
    • LocalizedTagSupport - furtherst curam class in the tag ancestor chain, adds locale, timezone, text direction awareness to all the curam tags.
    • Parameterised - helper interface to add parameter tag support, requires a method in the parent tag to add the named parameters from the children (nested parameter, say linParameter tags; might be removed when all the parameter tags consumers are converted into renderers or equivalent.
    • ScreenContext - support for the special request parameter which might also shape the particular tag output. This one would not go, since it isn't used by the tag infrastructure only, however it might be moved from the taglib to a more relevant location eventually.
    • SetConditionalDisplayValue - utility for implementing the CONDITION tag and the conditional display, sets or updates page context value to be consumed by the subsequent CONDITION tags on a page to render or skip part of the page layout.
    • TagLibUtil - another utility class, outside the tag inheritance chain but providing tags with the services like code table code and frequency pattern translation, current loop index and domain information. Code base should be scanned to see if this is used outside the tag library by some other parts of infrastructure.

 Component Renderers

Component renderers are supposed to be tabbed app "furniture" building blocks or layout elements and not related with the user data or domains; the tabbed application "furniture" components do not belong to any of the single pages (content) and do not match any tags.

HOWEVER, in practice as the following shows they are the mix of the tags which can be used on a single page and even bound to the domains (see the analysis below). Therefore it is felt that the classification and the resulting package structure might need a review and adjustments going forward.

The following classification is just the result of the first glimpse and might be refined.
